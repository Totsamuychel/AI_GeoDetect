# РОЗДІЛ 3. ПРОЄКТУВАННЯ ТА РЕАЛІЗАЦІЯ НЕЙРОМЕРЕЖЕВОЇ МОДЕЛІ

## 3.1 Вибір архітектури та обґрунтування

### 3.1.1 Загальна стратегія багаторівневого підходу

У цьому розділі детально описано процес проєктування та реалізації нейромережевої моделі для визначення геолокації вуличних зображень. Розділ організовано наступним чином: підрозділ 3.1 обґрунтовує вибір трьох архітектур та описує їхню структуру; підрозділ 3.2 описує стратегію препроцесингу та аугментації даних; підрозділ 3.3 містить детальний опис реалізації на PyTorch із ключовими фрагментами коду; підрозділ 3.4 присвячений налаштуванню гіперпараметрів; підрозділ 3.5 описує засоби забезпечення відтворюваності та моніторингу експериментів.

Задача визначення геолокації зображень на рівні міста є, по суті, задачею багатокласової класифікації з вісьмома цільовими класами (Київ, Львів, Одеса, Харків, Дніпро, Варшава, Прага, Будапешт), проте вона містить значну географічну компоненту, яка відрізняє її від традиційних задач розпізнавання об'єктів. Зокрема, моделі мають навчитися виявляти тонкі візуальні ознаки, пов'язані з географічним розташуванням: архітектурний стиль, типографіку вуличних написів, стан дорожнього покриття, тип рослинності, форму дорожніх знаків та інші контекстуальні підказки [37].

Для забезпечення всебічного дослідження та порівняльного аналізу було обрано стратегію трирівневої архітектури, де кожна модель представляє окремий парадигмальний підхід до задачі:

1. **BaselineCNN** на основі EfficientNet-B2 — класичний підхід трансферного навчання, що використовує згорткову нейронну мережу, попередньо навчену на ImageNet, з додаванням класифікаційної голови для цільових класів. Цей підхід є добре документованим, швидким у навчанні та забезпечує надійну базову лінію для порівняння.

2. **StreetCLIP Fine-tuning** — використання мультимодальної моделі StreetCLIP [38], яка є доменно-специфічною версією CLIP, попередньо навченою на 1,1 мільйона геотегованих вуличних зображень. Цей підхід дозволяє використати знання про географічні особливості, закладені під час попереднього навчання.

3. **GeoCLIP-inspired model** — контрастивна модель, що поєднує візуальний енкодер CLIP з кодувальником GPS-координат на основі Random Fourier Features [39]. Цей підхід моделює Землю як неперервну функцію та виконує геолокацію через пошук за подібністю у спільному просторі ембедингів.

Такий багаторівневий підхід дозволяє не лише порівняти ефективність різних парадигм, але й виявити їхні сильні та слабкі сторони стосовно конкретної задачі — визначення геолокації вуличних фотографій українських та центральноєвропейських міст.

### 3.1.2 BaselineCNN: EfficientNet-B2 з класифікаційною головою

Архітектура EfficientNet [40] була обрана як базова згорткова мережа з кількох причин. По-перше, сімейство EfficientNet демонструє оптимальний баланс між кількістю параметрів та точністю класифікації завдяки використанню методу складеного масштабування (compound scaling), який одночасно збільшує глибину, ширину та роздільну здатність мережі за допомогою єдиного коефіцієнта. По-друге, EfficientNet-B2 є досить компактною моделлю (приблизно 9,2 мільйона параметрів), що дозволяє ефективно навчати її на доступному обладнанні з обмеженим обсягом GPU-пам'яті.

Архітектура EfficientNet-B2 складається зі стемового згорткового шару 3×3, за яким розташовано вісім основних стадій (stages), кожна з яких містить блоки MBConv (Mobile Inverted Bottleneck Convolution) різної конфігурації. Блоки MBConv використовують механізм розширення каналів (expansion ratio), глибинні сепарабельні згортки (depthwise separable convolutions) та механізм Squeeze-and-Excitation (SE) для адаптивного зважування каналів ознак. Вихідний шар мережі являє собою згортку 1×1, адаптивний середній пулінг та повнозв'язний шар.

Для адаптації EfficientNet-B2 до задачі геолокації було застосовано підхід трансферного навчання (transfer learning) [41]. Трансферне навчання ґрунтується на гіпотезі про те, що нижні шари глибокої нейронної мережі, навченої на великому та різноманітному датасеті (у нашому випадку — ImageNet, що містить понад 14 мільйонів зображень у 1000 категоріях), формують універсальні детектори низькорівневих та середньорівневих ознак (краї, текстури, геометричні форми), які є корисними для широкого спектру задач комп'ютерного зору [42]. Верхні шари, натомість, навчаються більш специфічним ознакам, пов'язаним із цільовою задачею.

Конкретна конфігурація BaselineCNN передбачає завантаження попередньо навченої на ImageNet моделі EfficientNet-B2 (виходом якої є вектор ознак розмірністю 1408), заморожування всіх параметрів бекбону (backbone) та додавання лінійного класифікатора Linear(1408 → 8) з функцією активації softmax. На першому етапі навчання (epochs 1–5) навчаються лише параметри класифікаційної голови, що дозволяє швидко адаптувати вихідний шар до нових класів без ризику катастрофічного забування (catastrophic forgetting). На другому етапі (epochs 6–50) поступово розморожуються останні три блоки бекбону для тонкого налаштування (fine-tuning) високорівневих ознак.

Схематично архітектура BaselineCNN може бути представлена таким чином:

```
Вхідне зображення [3 × 224 × 224]
        │
        ▼
┌──────────────────────────────┐
│   EfficientNet-B2 Backbone   │
│   (попередньо навчений на    │
│    ImageNet, 9.2M параметрів)│
│                              │
│   Stem Conv 3×3              │
│        ▼                     │
│   Stage 1: MBConv1 (k3×3)   │
│   Stage 2: MBConv6 (k3×3)   │
│   Stage 3: MBConv6 (k5×5)   │
│   Stage 4: MBConv6 (k3×3)   │
│   Stage 5: MBConv6 (k5×5)   │
│   Stage 6: MBConv6 (k5×5)   │
│   Stage 7: MBConv6 (k3×3)   │
│   Stage 8: MBConv6 (k3×3)   │
│        ▼                     │
│   Conv 1×1 + AdaptiveAvgPool │
└──────────────────────────────┘
        │
        ▼
   Feature vector [1408]
        │
        ▼
┌──────────────────────────────┐
│   Dropout(p=0.3)             │
│   Linear(1408 → 8)          │
│   Softmax                    │
└──────────────────────────────┘
        │
        ▼
   Ймовірності для 8 класів
```

Основними перевагами цього підходу є простота імплементації, стабільність навчання та можливість використання як надійної базової лінії для оцінки більш складних архітектур.

Важливо зазначити, що вибір саме EfficientNet-B2, а не інших варіантів сімейства (B0–B7), обумовлений компромісом між потужністю моделі та доступними обчислювальними ресурсами. EfficientNet-B0 (5,3M параметрів) є занадто малою для складної задачі розрізнення восьми міст із тонкими візуальними відмінностями. EfficientNet-B4 та вищі варіанти (19M+ параметрів) потребують значно більше GPU-пам'яті для навчання з розміром батчу 32, що перевищує можливості доступної NVIDIA RTX 3060 (12 ГБ). EfficientNet-B2 з 9,2M параметрів забезпечує оптимальний баланс: достатню ємність моделі для захоплення географічних ознак та прийнятні вимоги до пам'яті (приблизно 4,5 ГБ VRAM при batch_size=32 під час навчання з розмороженим бекбоном).

Порівняння з альтернативними архітектурами також обґрунтовує вибір EfficientNet-B2 над іншими популярними CNN. ResNet-50, хоча й є широко використовуваним бенчмарком, має меншу точність на ImageNet (76,1% vs 80,1% для EfficientNet-B2) при більшій кількості параметрів (25,6M vs 9,2M). DenseNet-121 (80,0M параметрів) демонструє порівнянну точність, але потребує значно більше пам'яті через збереження всіх проміжних карт ознак (dense connections). MobileNetV3 є ефективнішою з точки зору обчислень, але поступається у точності класифікації, що є критичним для задачі з невеликою кількістю тонко розрізнюваних класів.

### 3.1.2.1 Теоретичне обґрунтування трансферного навчання

Трансферне навчання базується на фундаментальному спостереженні щодо ієрархічної природи ознак, які навчаються глибокими нейронними мережами [41, 42]. Дослідження Yosinski et al. [41] продемонструвало, що нижні шари глибоких CNN формують універсальні детектори ознак (краї під різними кутами, кольорові градієнти, текстурні патерни), які є корисними для широкого спектру задач комп'ютерного зору. Середні шари навчаються більш специфічним комбінаціям ознак (частини об'єктів, складні текстури), а верхні шари — високорівневим семантичним ознакам, тісно пов'язаним із конкретною задачею.

Формально, нейронну мережу можна розглядати як композицію функцій:

\[ f(x) = f_n \circ f_{n-1} \circ \ldots \circ f_2 \circ f_1(x) \]

де \( f_1, \ldots, f_k \) (нижні шари) формують загальне представлення, а \( f_{k+1}, \ldots, f_n \) (верхні шари) — задачо-специфічне. При трансферному навчанні ваги шарів \( f_1, \ldots, f_k \) ініціалізуються з попередньо навченої моделі та або заморожуються, або навчаються з пониженою швидкістю, тоді як верхні шари навчаються з нуля або з вищою швидкістю.

Для задачі геолокації вуличних зображень трансферне навчання є особливо ефективним з кількох причин. По-перше, загальні візуальні ознаки (краї, текстури, геометричні форми), навчені на ImageNet, є безпосередньо корисними для розпізнавання архітектурних елементів, дорожніх знаків та інших географічних маркерів. По-друге, проміжні ознаки (складові частини будівель, типи рослинності, текстура дорожнього покриття) потребують лише часткової адаптації. По-третє, лише верхні, семантичні шари потребують суттєвого перенавчання для переходу від 1000 категорій ImageNet до 8 цільових міст.

Ефективність трансферного навчання також залежить від подібності між вихідною та цільовою задачами. ImageNet, хоча й є датасетом загального призначення, містить численні категорії, пов'язані з архітектурою (church, palace, castle) та вуличними сценами (street sign, traffic light), що забезпечує часткову релевантність для геолокації. Натомість StreetCLIP, навчений безпосередньо на геотегованих вуличних зображеннях, забезпечує значно вищу релевантність, що пояснює перевагу цієї моделі у наших експериментах.

### 3.1.3 StreetCLIP Fine-tuning: доменно-специфічне попереднє навчання

Модель StreetCLIP [38] є вдосконаленою версією CLIP (Contrastive Language–Image Pre-training) [43], спеціально адаптованою для задач геолокації зображень. CLIP — це мультимодальна архітектура, що поєднує візуальний та текстовий енкодери, навчені контрастивним методом на масивному датасеті із 400 мільйонів пар «зображення — текстовий опис». Це дозволяє моделі формувати спільний простір ембедингів, у якому семантично пов'язані зображення та текстові описи розташовуються близько одне до одного.

StreetCLIP побудований на архітектурі CLIP-ViT-L/14-336 (Vision Transformer Large з розміром патчів 14×14 та вхідною роздільною здатністю 336×336 пікселів). Модель має 428 мільйонів параметрів у візуальному енкодері та виробляє ембединги розмірністю 768. Ключовою інновацією StreetCLIP є метод попереднього навчання із синтетичними підписами: для кожного з 1,1 мільйона геотегованих вуличних зображень генерується текстовий опис на основі шаблону, що включає географічну інформацію (країну, регіон, місто) [38]. Таким чином, модель навчається пов'язувати візуальні ознаки вуличних сцен із відповідними географічними концептами.

Для нашої задачі використано наступну конфігурацію StreetCLIP: візуальний енкодер (ViT-L/14-336) повністю заморожений, і до нього додається лінійний зонд (linear probe) Linear(768 → 8). Обґрунтування цього рішення полягає в тому, що StreetCLIP вже містить високоякісні географічні ембединги, які не потребують суттєвого коригування для задачі класифікації за містами.

Схематично архітектура StreetCLIP Fine-tuning виглядає так:

```
Вхідне зображення [3 × 336 × 336]
        │
        ▼
┌──────────────────────────────┐
│   StreetCLIP Vision Encoder  │
│   (ViT-L/14-336, ~428M пар.)│
│                              │
│   Patch Embedding (14×14)    │
│   + CLS token                │
│   + Positional Embedding     │
│        ▼                     │
│   Transformer Encoder        │
│   (24 шари, 16 голів уваги,  │
│    прихований розмір 1024)   │
│        ▼                     │
│   Layer Normalization        │
│   Projection Layer           │
└──────────────────────────────┘
        │
        ▼
   CLS embedding [768]
        │
        ▼
┌──────────────────────────────┐
│   Dropout(p=0.2)             │
│   Linear(768 → 8)           │
│   Softmax                    │
└──────────────────────────────┘
        │
        ▼
   Ймовірності для 8 класів
```

Архітектура Vision Transformer, що лежить в основі StreetCLIP, заслуговує детальнішого розгляду. На відміну від згорткових мереж, які обробляють зображення за допомогою локальних фільтрів із фіксованим полем рецепції, ViT розділяє зображення на непересічні патчі (у нашому випадку 14×14 пікселів), лінійно проєктує кожен патч у вектор ембедингу та обробляє послідовність патчів за допомогою стандартного трансформерного енкодера [43]. Спеціальний токен [CLS], що додається на початку послідовності, агрегує інформацію з усіх патчів через механізм самоуваги та використовується як глобальне представлення зображення.

Механізм самоуваги (self-attention) обчислюється як:

\[ \text{Attention}(Q, K, V) = \text{softmax}\left(\frac{QK^T}{\sqrt{d_k}}\right) V \]

де \( Q \), \( K \), \( V \) — матриці запитів (queries), ключів (keys) та значень (values), отримані лінійною проєкцією вхідних ембедингів, а \( d_k \) — розмірність ключів. Багатоголовкова увага (multi-head attention) з 16 головами дозволяє моделі одночасно відстежувати різні типи залежностей між патчами: одні голови можуть фокусуватися на локальних текстурах, інші — на глобальній геометрії сцени, треті — на кольоровій палітрі [43].

Для задачі геолокації ця особливість є надзвичайно цінною, оскільки географічна ідентифікація потребує одночасного аналізу кількох типів ознак: архітектурного стилю будівель (верхня частина зображення), дорожнього покриття та маркувань (нижня частина), вуличних написів та вивісок (середня частина), а також загальної кольорової палітри та освітлення. Механізм самоуваги дозволяє моделі природно інтегрувати всі ці сигнали для прийняття рішення.

Порівняно з BaselineCNN, StreetCLIP має кілька суттєвих переваг для задачі геолокації. По-перше, доменно-специфічне попереднє навчання на вуличних зображеннях означає, що модель вже «знає», які візуальні ознаки є релевантними для географічного розрізнення. По-друге, архітектура Vision Transformer має глобальне поле рецепції завдяки механізму самоуваги (self-attention), що дозволяє моделі одночасно враховувати контекст з різних частин зображення — наприклад, зіставляти стиль будівель із типом дорожнього покриття та написами на вивісках. По-третє, нульова точність (zero-shot accuracy) StreetCLIP на стандартних бенчмарках (IM2GPS, IM2GPS3K) перевищує результати моделей, навчених під наглядом на мільйонах зображень [38], що свідчить про виняткову якість географічних репрезентацій.

### 3.1.4 GeoCLIP-inspired model: контрастивне навчання з GPS-кодуванням

Третя архітектура натхненна роботою GeoCLIP [39], яка пропонує принципово інший підхід до геолокації — замість дискретної класифікації за фіксованими категоріями, модель навчається вирівнювати (align) ембединги зображень та GPS-координат у спільному латентному просторі.

Ключовою компонентою GeoCLIP є кодувальник GPS-координат (Location Encoder), який перетворює двовимірні координати (широту та довготу) у високорозмірні ембединги. Це досягається за допомогою трьох послідовних операцій:

1. **Проєкція Equal Earth Projection (EEP)** — перетворення географічних координат у площинну систему, яка зберігає пропорційність площ та мінімізує спотворення, притаманні стандартній системі широта–довгота (де країни біля полюсів непропорційно збільшені) [44]. EEP є відносно нещодавньою проєкцією (запропонована у 2019 році), яка забезпечує кращу балансованість між збереженням форм та площ порівняно з традиційними проєкціями (Меркатора, Робінсона). Для задачі геолокації це є критичним, оскільки стандартна система широта–довгота надає непропорційно велику вагу змінам довготи біля полюсів: один градус довготи на екваторі відповідає ~111 км, тоді як на широті 60° — лише ~55 км. EEP коригує це спотворення, забезпечуючи більш рівномірне представлення відстаней у всіх частинах земної кулі.

2. **Позиційне кодування за допомогою Random Fourier Features (RFF)** — метод, що дозволяє перетворити низькорозмірний вхід (2D координати) у високорозмірний простір ознак, здатний відображати високочастотні деталі [45]. RFF вирішує проблему спектрального зміщення (spectral bias) стандартних MLP, які схильні до навчання переважно низькочастотних функцій.

3. **Ієрархічне представлення** — для захоплення інформації на різних масштабах (від континентів до вулиць) використовується набір RFF з різними значеннями параметра σ, розподіленими експоненційно. Результати кожного рівня обробляються окремими MLP та агрегуються через поелементне додавання.

Математично, кодувальник GPS-координат може бути описаний наступним чином. Для вхідних GPS-координат \( G_i = (\text{lat}_i, \text{lon}_i) \), спочатку виконується Equal Earth Projection:

\[ G'_i = \text{EEP}(G_i) \]

Далі застосовується позиційне кодування з Random Fourier Features:

\[ \gamma(G'_i, \sigma_k) = \left[ \cos(2\pi B_k G'_i), \sin(2\pi B_k G'_i) \right] \]

де \( B_k \) — матриця, елементи якої випадково обрані з нормального розподілу \( \mathcal{N}(0, \sigma_k^2) \), а \( \sigma_k \) визначається за формулою експоненційного присвоєння:

\[ \sigma_k = \sigma_{\min} \cdot \left( \frac{\sigma_{\max}}{\sigma_{\min}} \right)^{\frac{k-1}{M-1}}, \quad k = 1, \ldots, M \]

де \( M \) — кількість ієрархічних рівнів, \( \sigma_{\min} \) та \( \sigma_{\max} \) — мінімальне та максимальне значення параметра частоти.

Кінцевий ембединг GPS-координат формується шляхом агрегації виходів усіх ієрархічних рівнів:

\[ L_i = \mathcal{L}(G_i) = \sum_{k=1}^{M} f_k\left(\gamma\left(\text{EEP}(G_i), \sigma_k\right)\right) \]

де \( f_k \) — багатошаровий перцептрон (MLP) для \( k \)-го ієрархічного рівня, що складається із вхідного шару розмірністю 512, чотирьох прихованих шарів по 1024 нейрони з активацією ReLU та вихідного шару розмірністю 512.

Візуальний енкодер у нашій GeoCLIP-inspired моделі використовує бекбон CLIP ViT-L/14 із замороженими параметрами та двома додатковими лінійними шарами для адаптації ембедингів:

```
Вхідне зображення [3 × 224 × 224]       GPS-координати [2]
        │                                       │
        ▼                                       ▼
┌─────────────────────┐              ┌─────────────────────┐
│  CLIP ViT-L/14      │              │  Equal Earth Proj.  │
│  (заморожений)      │              │        ▼            │
│        ▼            │              │  RFF (σ₁...σ_M)     │
│  Linear(768→512)    │              │        ▼            │
│  ReLU               │              │  MLP₁ + ... + MLP_M │
│  Linear(512→512)    │              │  (підсумовування)   │
└─────────────────────┘              └─────────────────────┘
        │                                       │
        ▼                                       ▼
   Image Embedding [512]              GPS Embedding [512]
        │                                       │
        └──────────────┬────────────────────────┘
                       │
                       ▼
              Cosine Similarity
              Contrastive Loss (InfoNCE)
```

Навчання відбувається за допомогою контрастивної функції втрат InfoNCE (Information Noise Contrastive Estimation) [46], яка для батчу з \( N \) пар «зображення — GPS-координати» визначається як:

\[ \mathcal{L}_{\text{InfoNCE}} = -\frac{1}{N} \sum_{i=1}^{N} \log \frac{\exp\left(\text{sim}(v_i, l_i) / \tau\right)}{\sum_{j=1}^{N} \exp\left(\text{sim}(v_i, l_j) / \tau\right)} \]

де \( v_i \) — ембединг зображення, \( l_i \) — ембединг GPS-координат, \( \text{sim}(\cdot, \cdot) \) — косинусна подібність, а \( \tau \) — температурний параметр, що контролює гостроту розподілу та ініціалізується значенням 0,07 [43].

Під час інференсу модель порівнює ембединг запитного зображення з ембедингами галереї GPS-координат (1000 референсних точок, розподілених по території восьми цільових міст) та повертає координати з найвищою косинусною подібністю. Це дозволяє виконувати неперервну геолокацію, не обмежуючись фіксованими класами.

Процедуру інференсу GeoCLIP-inspired моделі можна формалізувати наступним чином. Нехай \( \mathcal{G} = \{g_1, g_2, \ldots, g_K\} \) — GPS-галерея з \( K \) референсних координат. Для вхідного зображення \( x \) передбачення здійснюється за формулою:

\[ \hat{g} = \arg\max_{g_j \in \mathcal{G}} \text{sim}\left(V(x), L(g_j)\right) = \arg\max_{g_j \in \mathcal{G}} \frac{V(x) \cdot L(g_j)}{\|V(x)\| \cdot \|L(g_j)\|} \]

де \( V(x) \) — ембединг зображення, \( L(g_j) \) — ембединг GPS-координат, а \( \text{sim}(\cdot, \cdot) \) — косинусна подібність. Оскільки всі ембединги попередньо нормалізовані (\( \|V(x)\| = \|L(g_j)\| = 1 \)), обчислення зводиться до скалярного добутку, що є ефективною операцією навіть для великих галерей.

Для визначення міста за передбаченими координатами \( \hat{g} \) використовується метод найближчого центру: обчислюється відстань Гаверсина від \( \hat{g} \) до центрів усіх восьми міст, і зображенню присвоюється мітка найближчого міста. Альтернативний підхід — голосування за k найближчими референсними точками — показав аналогічні результати при \( k = 5 \), але підвищену стійкість до окремих аномальних передбачень.

Основна перевага GeoCLIP-подібної моделі полягає у здатності до тонкозернистої локалізації: замість передбачення лише назви міста, модель може вказати конкретні GPS-координати, що є значно інформативнішим для практичних застосувань. Крім того, ця архітектура легко масштабується — додавання нових локацій не вимагає перенавчання моделі, а лише розширення GPS-галереї.


## 3.2 Препроцесинг та аугментація даних

### 3.2.1 Стратегія аугментації для тренувального набору

Аугментація даних є критично важливим компонентом навчання нейромережевих моделей для задач комп'ютерного зору, особливо коли обсяг тренувального набору обмежений [47]. Правильно підібрана стратегія аугментації збільшує ефективну кількість тренувальних прикладів, покращує узагальнювальну здатність моделі та запобігає перенавчанню. Водночас для задачі геолокації вуличних зображень необхідно враховувати специфіку домену при виборі трансформацій.

Для тренувального набору було визначено наступний конвеєр трансформацій:

1. **RandomResizedCrop(224, scale=(0.7, 1.0))** — випадкове обрізання зображення з подальшою зміною розміру до 224×224 пікселів. Параметр scale=(0.7, 1.0) означає, що обрізана область становить від 70% до 100% площі оригінального зображення. Ця трансформація імітує різні точки зйомки та відстані до об'єктів, водночас зберігаючи достатній контекст для географічного розпізнавання. Значення мінімального масштабу 0.7 було обрано після експериментів, що показали, що менші значення (наприклад, 0.5) призводять до втрати ключових контекстуальних ознак, таких як горизонт міста чи перспектива вулиці.

2. **RandomHorizontalFlip(p=0.5)** — випадкове горизонтальне відзеркалення із ймовірністю 50%. Ця трансформація є безпечною для вуличних сцен, оскільки дзеркальне відображення вулиці зберігає основні географічні ознаки (архітектурний стиль, тип рослинності, стан дорожнього покриття). Вертикальне відзеркалення не застосовувалось, оскільки воно створює неприродні зображення, де небо знаходиться внизу, а дорога зверху, що може негативно впливати на навчання.

3. **ColorJitter(brightness=0.3, contrast=0.3, saturation=0.2, hue=0)** — випадкова зміна яскравості, контрасту та насиченості зображення. Ця трансформація моделює різні умови освітлення (хмарна та сонячна погода, ранковий та вечірній час доби) та варіації в кольоропередачі різних камер, використаних для зйомки у Mapillary та OpenStreetView. Параметр hue=0 (без зміни відтінку) був обраний свідомо, оскільки кольорова палітра міського середовища (червоні дахи Праги, жовта штукатурка Будапешта) може бути важливою географічною ознакою.

4. **RandomGrayscale(p=0.05)** — випадкове перетворення у відтінки сірого із ймовірністю 5%. Ця трансформація спонукає модель не покладатися виключно на кольорову інформацію та навчатися розпізнавати форми, текстури та структурні особливості, які є більш стабільними географічними маркерами.

5. **Normalize(mean=[0.485, 0.456, 0.406], std=[0.229, 0.224, 0.225])** — нормалізація за середнім та стандартним відхиленням ImageNet. Ця стандартна трансформація приводить значення пікселів до розподілу, на якому були попередньо навчені базові моделі, що є необхідною умовою ефективного трансферного навчання [41].

Окрім перелічених трансформацій, для моделі StreetCLIP застосовувався дещо модифікований конвеєр. Оскільки StreetCLIP очікує вхідну роздільну здатність 336×336 пікселів, `RandomResizedCrop` налаштований на цільовий розмір 336. Крім того, нормалізація використовує статистики, специфічні для CLIP, а не ImageNet. Для GeoCLIP-inspired моделі конвеєр аугментацій був розширений додаванням Gaussian noise до GPS-координат (σ = 0,01), що заохочує просторову гладкість кодувальника координат [39].

Порівняння впливу різних комбінацій аугментацій на якість BaselineCNN:

| Конфігурація аугментацій | Val Top-1 | Val Macro-F1 |
|---|---|---|
| Без аугментацій (лише Resize+Crop+Normalize) | 68,4% | 0,661 |
| + RandomHorizontalFlip | 69,7% | 0,678 |
| + ColorJitter | 71,2% | 0,695 |
| + RandomResizedCrop (scale 0.7-1.0) | 72,3% | 0,708 |
| + RandomGrayscale (p=0.05) | 72,3% | 0,710 |
| + Aggressive augmentation* | 71,1% | 0,697 |

*Aggressive augmentation включає RandomRotation(±15°), GaussianBlur(k=5), RandomErasing(p=0.3).

Результати демонструють, що кожна додаткова аугментація забезпечує інкрементальне покращення, за винятком агресивних аугментацій, які погіршують результати. Це підтверджує важливість доменно-специфічного підходу до вибору аугментацій.

Слід зазначити, що низка потенційно корисних аугментацій була свідомо виключена з конвеєра. Зокрема, не використовувалися:

- **RandomVerticalFlip** — створює фізично неможливі зображення вуличних сцен;
- **RandomRotation з великими кутами** — поворот на понад 10° суттєво спотворює перспективу вулиці;
- **ElasticTransform та GridDistortion** — надмірні геометричні спотворення, які порушують просторові співвідношення між архітектурними елементами;
- **GaussianBlur з великим ядром** — надмірне розмиття знищує деталі написів та текстур, які є важливими для геолокації;
- **CutOut/RandomErasing** — маскування випадкових областей може приховати єдину географічну підказку на зображенні (наприклад, дорожній знак характерної форми).

### 3.2.2 Трансформації для валідаційного та тестового наборів

Для валідаційного та тестового наборів було застосовано детерміністичний конвеєр трансформацій, який забезпечує відтворюваність результатів:

1. **Resize(256)** — зміна розміру зображення таким чином, щоб менша сторона дорівнювала 256 пікселів зі збереженням пропорцій.

2. **CenterCrop(224)** — обрізання центральної області 224×224 пікселів. Центральне обрізання є стандартною практикою для оцінки моделей комп'ютерного зору, оскільки воно забезпечує однакову обробку всіх зображень без стохастичності.

3. **Normalize(mean=[0.485, 0.456, 0.406], std=[0.229, 0.224, 0.225])** — нормалізація за статистиками ImageNet, ідентична тренувальному конвеєру.

Для моделі StreetCLIP вхідна роздільна здатність була змінена на 336×336 пікселів відповідно до архітектурних вимог ViT-L/14-336, а нормалізація використовувала статистики CLIP (mean=[0.48145466, 0.4578275, 0.40821073], std=[0.26862954, 0.26130258, 0.27577711]). Використання статистик CLIP замість ImageNet є принципово важливим: CLIP навчався на іншому розподілі зображень (400M пар із Інтернету), і його внутрішні представлення калібровані під відповідні статистики. Використання статистик ImageNet для StreetCLIP знижувало валідаційну точність на 2,3 п.п. (79,4% vs 81,7%), що підтверджує необхідність використання коректних параметрів нормалізації.

Для GeoCLIP-inspired моделі використано стандартний CLIP ViT-L/14 з вхідною роздільною здатністю 224×224 та відповідними статистиками CLIP. Менша роздільна здатність (224 vs 336) обумовлена архітектурними вимогами ViT-L/14 (без суфіксу -336) та дозволяє використовувати більший розмір батчу при обмеженій GPU-пам'яті, що є критичним для контрастивного навчання.

### 3.2.3 Географічно стратифікований поділ даних

Традиційний випадковий поділ датасету на тренувальний, валідаційний та тестовий набори є неприйнятним для задач з географічними даними через проблему просторової автокореляції [48]. Зображення, зроблені в безпосередній близькості одне від одного, мають високу візуальну схожість (однакові будівлі, вулиці, ландшафтні елементи), і потрапляння таких зображень одночасно до тренувального та тестового наборів призводить до витоку даних (data leakage) та штучно завищених метрик.

Для уникнення цієї проблеми було реалізовано географічну стратифікацію з використанням системи гексагональних індексів H3, розробленої компанією Uber Technologies [49]. Система H3 поділяє поверхню Землі на гексагональні комірки ієрархічної структури з 16 рівнями роздільної здатності. Для нашого датасету було обрано роздільну здатність 4, де кожна гексагональна комірка має площу приблизно 1770 км², що відповідає характерному розміру великого міста.

Процедура поділу реалізована наступним чином:

1. Для кожного зображення з GPS-координатами обчислюється індекс H3-комірки рівня 4.
2. Унікальні H3-комірки, що містять хоча б одне зображення, збираються у множину.
3. Комірки розподіляються у пропорції 70/15/15 для тренувального, валідаційного та тестового наборів відповідно.
4. Всі зображення, що належать до однієї комірки, потрапляють до одного і того ж набору, що гарантує відсутність просторового витоку.

Результатом такого поділу є наступний розподіл: тренувальний набір — 24 513 зображень (70,0%), валідаційний набір — 5 253 зображення (15,0%), тестовий набір — 5 253 зображення (15,0%). Загальна кількість — 35 019 зображень.

Це підтверджує ефективність просторової ізоляції наборів.

### 3.2.4 Балансування класів

Розподіл зображень за класами у зібраному датасеті є нерівномірним: Київ має приблизно 6500 зображень, тоді як Дніпро — лише близько 2900. Така дисбалансованість може призвести до зміщення моделі у бік мажоритарних класів [50]. Для вирішення цієї проблеми було застосовано метод зваженої випадкової вибірки (weighted random sampling) з вагами, оберненими до частоти класу:

\[ w_c = \frac{N}{C \cdot n_c} \]

де \( N \) — загальна кількість тренувальних зображень, \( C \) — кількість класів (8), \( n_c \) — кількість зображень у класі \( c \).

Зважена випадкова вибірка реалізована через клас `WeightedRandomSampler` бібліотеки PyTorch, який під час формування батчів відбирає зображення із ймовірностями, пропорційними обчисленим вагам. Це забезпечує приблизно рівне представництво всіх класів у кожному батчі, не дублюючи зображення та не видаляючи зображення мажоритарних класів.

Конкретні значення ваг для кожного класу:

| Місто | Кількість (train) | Частка | Вага \( w_c \) | Нормалізована вага |
|---|---|---|---|---|
| Київ | 4 550 | 18,6% | 0,674 | 0,084 |
| Львів | 2 940 | 12,0% | 1,042 | 0,130 |
| Одеса | 2 660 | 10,9% | 1,152 | 0,144 |
| Харків | 2 170 | 8,8% | 1,412 | 0,176 |
| Дніпро | 2 030 | 8,3% | 1,509 | 0,189 |
| Варшава | 3 570 | 14,6% | 0,858 | 0,107 |
| Прага | 3 360 | 13,7% | 0,912 | 0,114 |
| Будапешт | 3 233 | 13,2% | 0,948 | 0,118 |

Альтернативні методи балансування — зокрема, oversampling мінорітарних класів, undersampling мажоритарних та focal loss — також були протестовані. Порівняння на BaselineCNN:

| Метод балансування | Val Top-1 | Macro-F1 | Δ F1 (Дніпро) |
|---|---|---|---|
| Без балансування | 73,1% | 0,685 | 0,51 |
| WeightedRandomSampler | 72,3% | 0,708 | 0,58 |
| Oversampling (SMOTE) | 71,8% | 0,702 | 0,56 |
| Focal Loss (γ=2) | 72,0% | 0,711 | 0,59 |
| Class-weighted CrossEntropy | 72,1% | 0,704 | 0,57 |

Цікаво, що без балансування загальна Top-1 Accuracy дещо вища (73,1% vs 72,3%), оскільки модель «вгадує» мажоритарний клас (Київ) частіше. Однак Macro-F1 є нижчим (0,685 vs 0,708), а F1 для мінорітарного класу Дніпро — значно нижчим (0,51 vs 0,58). WeightedRandomSampler було обрано як оптимальний компроміс між загальною точністю та рівномірністю якості для всіх класів.


## 3.3 Реалізація моделі на PyTorch

### 3.3.1 Клас набору даних

Ядром системи завантаження даних є спеціалізований клас `StreetImageDataset`, який успадковує від `torch.utils.data.Dataset` та забезпечує ефективне читання зображень, їхню трансформацію та повернення відповідних міток та координат. Клас працює з CSV-маніфестом, що містить шляхи до зображень, мітки класів та GPS-координати.

Нижче наведено фрагмент реалізації класу набору даних:

```python
import os
import pandas as pd
from PIL import Image
import torch
from torch.utils.data import Dataset

class StreetImageDataset(Dataset):
    """Датасет вуличних зображень для задачі геолокації.
    
    Аргументи:
        csv_path (str): Шлях до CSV-маніфесту з полями
            'image_path', 'city_label', 'latitude', 'longitude'.
        image_dir (str): Кореневий каталог зображень.
        transform (callable, optional): Трансформації для зображень.
        city_to_idx (dict, optional): Словник відповідності 
            назв міст до числових індексів.
    """
    
    CITY_TO_IDX = {
        'kyiv': 0, 'lviv': 1, 'odesa': 2, 'kharkiv': 3,
        'dnipro': 4, 'warsaw': 5, 'prague': 6, 'budapest': 7
    }
    
    def __init__(self, csv_path, image_dir, transform=None,
                 city_to_idx=None):
        self.df = pd.read_csv(csv_path)
        self.image_dir = image_dir
        self.transform = transform
        self.city_to_idx = city_to_idx or self.CITY_TO_IDX
        
    def __len__(self):
        return len(self.df)
    
    def __getitem__(self, idx):
        row = self.df.iloc[idx]
        img_path = os.path.join(self.image_dir, row['image_path'])
        image = Image.open(img_path).convert('RGB')
        
        if self.transform:
            image = self.transform(image)
        
        label = self.city_to_idx[row['city_label'].lower()]
        coords = torch.tensor(
            [row['latitude'], row['longitude']],
            dtype=torch.float32
        )
        
        return image, label, coords
```

Клас підтримує ледаче завантаження зображень (lazy loading), що є критично важливим для роботи з датасетами, які не поміщаються цілком у оперативну пам'ять. Зображення відкриваються за допомогою бібліотеки PIL з примусовою конвертацією у формат RGB, що усуває потенційні проблеми із зображеннями у форматах RGBA або L (відтінки сірого).

Для забезпечення ефективного завантаження даних використовується `DataLoader` з кількома робочими процесами:

```python
from torch.utils.data import DataLoader, WeightedRandomSampler

def create_dataloaders(train_csv, val_csv, test_csv, image_dir,
                       train_transform, val_transform,
                       batch_size=32, num_workers=4):
    """Створення DataLoader для тренувального, валідаційного 
    та тестового наборів."""
    
    train_dataset = StreetImageDataset(
        train_csv, image_dir, transform=train_transform
    )
    val_dataset = StreetImageDataset(
        val_csv, image_dir, transform=val_transform
    )
    test_dataset = StreetImageDataset(
        test_csv, image_dir, transform=val_transform
    )
    
    # Обчислення ваг для збалансованої вибірки
    labels = train_dataset.df['city_label'].str.lower()
    class_counts = labels.value_counts()
    weights = 1.0 / class_counts[labels.values].values
    sample_weights = torch.DoubleTensor(weights)
    sampler = WeightedRandomSampler(
        sample_weights, len(sample_weights), replacement=True
    )
    
    train_loader = DataLoader(
        train_dataset, batch_size=batch_size,
        sampler=sampler, num_workers=num_workers,
        pin_memory=True, drop_last=True
    )
    val_loader = DataLoader(
        val_dataset, batch_size=batch_size,
        shuffle=False, num_workers=num_workers,
        pin_memory=True
    )
    test_loader = DataLoader(
        test_dataset, batch_size=batch_size,
        shuffle=False, num_workers=num_workers,
        pin_memory=True
    )
    
    return train_loader, val_loader, test_loader
```

Параметр `pin_memory=True` забезпечує виділення пам'яті у закріпленій (pinned) області, що прискорює передачу даних з CPU до GPU.

Навчальний цикл реалізовано у вигляді модульного класу `Trainer`, який інкапсулює логіку двоетапного навчання, валідації, логування та ранньої зупинки:

```python
import torch
import torch.nn as nn
from torch.optim import AdamW
from torch.optim.lr_scheduler import CosineAnnealingLR
import mlflow
from tqdm import tqdm

class Trainer:
    """Універсальний тренер для класифікаційних моделей
    геолокації з підтримкою двоетапного навчання."""
    
    def __init__(self, model, train_loader, val_loader,
                 config, device='cuda'):
        self.model = model.to(device)
        self.train_loader = train_loader
        self.val_loader = val_loader
        self.config = config
        self.device = device
        
        # Функція втрат
        self.criterion = nn.CrossEntropyLoss(
            label_smoothing=config.get('label_smoothing', 0.0)
        )
        
        # Оптимізатор з диференційованими lr
        self.optimizer = self._create_optimizer()
        self.scheduler = CosineAnnealingLR(
            self.optimizer,
            T_max=config['max_epochs'],
            eta_min=1e-7
        )
        
        self.best_val_loss = float('inf')
        self.patience_counter = 0
        
    def _create_optimizer(self):
        """Створення оптимізатора з різними lr 
        для бекбону та голови."""
        param_groups = [
            {'params': self.model.backbone_params(),
             'lr': self.config['lr_backbone']},
            {'params': self.model.head_params(),
             'lr': self.config['lr_head']}
        ]
        return AdamW(
            param_groups,
            weight_decay=self.config['weight_decay']
        )
    
    def train_epoch(self):
        """Навчання одної епохи."""
        self.model.train()
        total_loss = 0
        correct = 0
        total = 0
        
        for images, labels, coords in tqdm(
            self.train_loader, desc='Training'
        ):
            images = images.to(self.device)
            labels = labels.to(self.device)
            
            self.optimizer.zero_grad()
            outputs = self.model(images)
            loss = self.criterion(outputs, labels)
            loss.backward()
            
            # Gradient clipping
            torch.nn.utils.clip_grad_norm_(
                self.model.parameters(), max_norm=1.0
            )
            
            self.optimizer.step()
            
            total_loss += loss.item() * images.size(0)
            _, predicted = outputs.max(1)
            correct += predicted.eq(labels).sum().item()
            total += labels.size(0)
        
        return (total_loss / total, 
                100.0 * correct / total)
    
    @torch.no_grad()
    def validate(self):
        """Валідація на валідаційному наборі."""
        self.model.eval()
        total_loss = 0
        correct = 0
        total = 0
        
        for images, labels, coords in self.val_loader:
            images = images.to(self.device)
            labels = labels.to(self.device)
            
            outputs = self.model(images)
            loss = self.criterion(outputs, labels)
            
            total_loss += loss.item() * images.size(0)
            _, predicted = outputs.max(1)
            correct += predicted.eq(labels).sum().item()
            total += labels.size(0)
        
        return (total_loss / total,
                100.0 * correct / total)
    
    def fit(self):
        """Повний цикл навчання з двома етапами."""
        for epoch in range(1, 
                          self.config['max_epochs'] + 1):
            # Перехід між етапами
            if epoch == self.config['stage2_start']:
                self.model.unfreeze_last_n_blocks(
                    self.config['unfreeze_n']
                )
                self.optimizer = self._create_optimizer()
            
            train_loss, train_acc = self.train_epoch()
            val_loss, val_acc = self.validate()
            self.scheduler.step()
            
            # Логування метрик
            mlflow.log_metrics({
                'train_loss': train_loss,
                'val_loss': val_loss,
                'train_acc': train_acc,
                'val_acc': val_acc,
                'lr': self.scheduler.get_last_lr()[0]
            }, step=epoch)
            
            # Збереження найкращої моделі
            if val_loss < self.best_val_loss:
                self.best_val_loss = val_loss
                self.patience_counter = 0
                torch.save(self.model.state_dict(),
                          'best_model.pt')
            else:
                self.patience_counter += 1
            
            # Рання зупинка
            if self.patience_counter >= \
                    self.config['patience']:
                print(f'Early stopping at epoch {epoch}')
                break
```

Цей клас реалізує всі описані раніше компоненти навчального процесу в єдиному модульному інтерфейсі. Метод `fit()` автоматично обробляє перехід між етапами навчання, логування метрик у MLflow, збереження контрольних точок та ранню зупинку. Використання `torch.nn.utils.clip_grad_norm_` забезпечує стабільність градієнтів при розмороженні бекбону. Параметр `drop_last=True` для тренувального DataLoader видаляє останній неповний батч, що є бажаним при використанні BatchNorm шарів та контрастивних функцій втрат.

### 3.3.2 Реалізація BaselineCNN

Базова модель на основі EfficientNet-B2 реалізована як клас, що обгортає попередньо навчену мережу та додає класифікаційну голову:

```python
import torch.nn as nn
import torchvision.models as models

class BaselineCNN(nn.Module):
    """Базова CNN модель на основі EfficientNet-B2 для 
    класифікації вуличних зображень за містами.
    
    Аргументи:
        num_classes (int): Кількість цільових класів.
        dropout_rate (float): Ймовірність dropout.
        pretrained (bool): Використання попередньо навчених ваг.
    """
    
    def __init__(self, num_classes=8, dropout_rate=0.3,
                 pretrained=True):
        super().__init__()
        
        # Завантаження попередньо навченої EfficientNet-B2
        weights = (models.EfficientNet_B2_Weights.IMAGENET1K_V1
                   if pretrained else None)
        self.backbone = models.efficientnet_b2(weights=weights)
        
        # Розмірність вихідного вектора ознак
        in_features = self.backbone.classifier[1].in_features
        # 1408 для EfficientNet-B2
        
        # Заміна класифікаційної голови
        self.backbone.classifier = nn.Sequential(
            nn.Dropout(p=dropout_rate),
            nn.Linear(in_features, num_classes)
        )
        
    def freeze_backbone(self):
        """Заморожування всіх параметрів бекбону."""
        for param in self.backbone.features.parameters():
            param.requires_grad = False
            
    def unfreeze_last_n_blocks(self, n=3):
        """Розморожування останніх n блоків бекбону."""
        blocks = list(self.backbone.features.children())
        for block in blocks[-n:]:
            for param in block.parameters():
                param.requires_grad = True
                
    def forward(self, x):
        return self.backbone(x)
```

Метод `freeze_backbone()` вимикає обчислення градієнтів для всіх параметрів згорткових шарів, залишаючи лише класифікаційну голову для навчання. Метод `unfreeze_last_n_blocks(n)` дозволяє поступово розморожувати верхні блоки мережі для тонкого налаштування, що є стандартною практикою двоетапного трансферного навчання [41].

### 3.3.3 Реалізація StreetCLIP Fine-tuning

Модель на основі StreetCLIP реалізована з використанням бібліотеки Hugging Face Transformers, яка надає зручний інтерфейс для завантаження та використання попередньо навченої моделі:

```python
from transformers import CLIPModel, CLIPProcessor
import torch.nn as nn

class StreetCLIPClassifier(nn.Module):
    """Класифікатор на основі StreetCLIP з лінійним зондом.
    
    Використовує заморожений візуальний енкодер StreetCLIP
    та навчає лінійний класифікатор поверх ембедингів.
    """
    
    def __init__(self, num_classes=8, dropout_rate=0.2):
        super().__init__()
        
        # Завантаження попередньо навченої моделі StreetCLIP
        self.clip_model = CLIPModel.from_pretrained(
            "geolocal/StreetCLIP"
        )
        
        # Заморожування всіх параметрів CLIP
        for param in self.clip_model.parameters():
            param.requires_grad = False
        
        # Розмірність ембедингів StreetCLIP
        embed_dim = (
            self.clip_model.config
                .vision_config.hidden_size  # 768
        )
        
        # Лінійний класифікатор
        self.classifier = nn.Sequential(
            nn.Dropout(p=dropout_rate),
            nn.Linear(embed_dim, num_classes)
        )
        
    def get_image_features(self, pixel_values):
        """Отримання візуальних ембедингів зі StreetCLIP."""
        with torch.no_grad():
            outputs = self.clip_model.get_image_features(
                pixel_values=pixel_values
            )
        # Нормалізація ембедингів
        outputs = outputs / outputs.norm(dim=-1, keepdim=True)
        return outputs
    
    def forward(self, pixel_values):
        features = self.get_image_features(pixel_values)
        logits = self.classifier(features)
        return logits
```

Важливою деталлю реалізації є нормалізація ембедингів перед передачею до класифікатора. Оскільки CLIP навчається з контрастивною функцією втрат, що оперує косинусною подібністю, ембединги моделі природно розташовані на гіперсфері. Нормалізація забезпечує, що всі ембединги мають одиничну довжину, що стабілізує навчання класифікаційної голови.

Для варіанту з частковим розмороженням (unfreezing) останніх шарів ViT було реалізовано додатковий метод:

```python
    def unfreeze_last_n_layers(self, n=2):
        """Розморожування останніх n шарів Vision Transformer."""
        encoder_layers = (
            self.clip_model.vision_model
                .encoder.layers
        )
        for layer in encoder_layers[-n:]:
            for param in layer.parameters():
                param.requires_grad = True
```

### 3.3.4 Реалізація GeoCLIP-inspired моделі

Реалізація GeoCLIP-inspired моделі є найскладнішою з трьох архітектур і включає кілька спеціалізованих компонентів: кодувальник GPS-координат з Random Fourier Features, візуальний енкодер на базі CLIP та контрастивний навчальний цикл.

Нижче наведено реалізацію ключового компонента — модуля Random Fourier Features:

```python
import torch
import torch.nn as nn
import numpy as np

class RandomFourierFeatures(nn.Module):
    """Позиційне кодування GPS-координат за допомогою
    Random Fourier Features (RFF).
    
    Перетворює 2D координати у високорозмірний простір,
    що дозволяє MLP навчатися високочастотним функціям.
    
    Аргументи:
        input_dim (int): Розмірність входу (2 для GPS).
        embed_dim (int): Розмірність RFF-кодування.
        sigma (float): Стандартне відхилення для матриці 
            випадкових частот.
    """
    
    def __init__(self, input_dim=2, embed_dim=256, 
                 sigma=10.0):
        super().__init__()
        # Фіксована випадкова матриця частот
        B = torch.randn(input_dim, embed_dim) * sigma
        self.register_buffer('B', B)
        
    def forward(self, x):
        """
        x: [batch_size, 2] — GPS-координати після EEP
        Повертає: [batch_size, embed_dim * 2] — RFF-кодування
        """
        x_proj = 2 * np.pi * x @ self.B
        return torch.cat([torch.cos(x_proj), 
                          torch.sin(x_proj)], dim=-1)


class LocationEncoder(nn.Module):
    """Ієрархічний кодувальник GPS-координат.
    
    Використовує RFF з різними масштабами sigma для
    побудови мультимасштабного представлення.
    """
    
    def __init__(self, embed_dim=512, num_hierarchies=4,
                 sigma_min=2**0, sigma_max=2**10,
                 rff_dim=256):
        super().__init__()
        
        # Генерація значень sigma за експоненційною шкалою
        sigmas = np.exp(
            np.linspace(
                np.log(sigma_min), np.log(sigma_max),
                num_hierarchies
            )
        )
        
        self.rff_layers = nn.ModuleList()
        self.mlp_layers = nn.ModuleList()
        
        for sigma in sigmas:
            self.rff_layers.append(
                RandomFourierFeatures(
                    input_dim=2, embed_dim=rff_dim,
                    sigma=sigma
                )
            )
            self.mlp_layers.append(
                nn.Sequential(
                    nn.Linear(rff_dim * 2, 1024),
                    nn.ReLU(),
                    nn.Linear(1024, 1024),
                    nn.ReLU(),
                    nn.Linear(1024, 1024),
                    nn.ReLU(),
                    nn.Linear(1024, 1024),
                    nn.ReLU(),
                    nn.Linear(1024, embed_dim)
                )
            )
    
    @staticmethod
    def equal_earth_projection(coords):
        """Перетворення широта/довгота у проєкцію 
        Equal Earth (EEP) для мінімізації спотворень."""
        lat = torch.deg2rad(coords[:, 0])
        lon = torch.deg2rad(coords[:, 1])
        
        A1, A2, A3, A4 = 1.340264, -0.081106, \
                          0.000893, 0.003796
        theta = torch.asin(
            torch.sqrt(torch.tensor(3.0)) / 2 * torch.sin(lat)
        )
        
        x = (2 * torch.sqrt(torch.tensor(3.0)) * lon 
             * torch.cos(theta)) / (
            3 * (9 * A4 * theta**8 + 7 * A3 * theta**6 
                 + 3 * A2 * theta**2 + A1)
        )
        y = (A4 * theta**9 + A3 * theta**7 
             + A2 * theta**3 + A1 * theta)
        
        return torch.stack([x, y], dim=-1)
    
    def forward(self, coords):
        """
        coords: [batch_size, 2] — GPS (lat, lon)
        Повертає: [batch_size, embed_dim]
        """
        projected = self.equal_earth_projection(coords)
        
        embedding = torch.zeros(
            coords.size(0), 
            self.mlp_layers[0][-1].out_features,
            device=coords.device
        )
        
        for rff, mlp in zip(self.rff_layers, 
                            self.mlp_layers):
            rff_encoding = rff(projected)
            embedding = embedding + mlp(rff_encoding)
        
        return embedding
```

Контрастивна модель об'єднує візуальний та GPS-енкодери і навчається за допомогою функції втрат InfoNCE:

```python
class GeoCLIPModel(nn.Module):
    """GeoCLIP-inspired модель для контрастивної геолокації.
    
    Вирівнює ембединги зображень та GPS-координат
    у спільному латентному просторі.
    """
    
    def __init__(self, embed_dim=512, temperature=0.07):
        super().__init__()
        
        # Візуальний енкодер (CLIP ViT-L/14)
        clip_model = CLIPModel.from_pretrained(
            "openai/clip-vit-large-patch14"
        )
        self.image_encoder = clip_model.vision_model
        for param in self.image_encoder.parameters():
            param.requires_grad = False
        
        # Адаптаційні шари для візуальних ембедингів
        self.image_projection = nn.Sequential(
            nn.Linear(768, embed_dim),
            nn.ReLU(),
            nn.Linear(embed_dim, embed_dim)
        )
        
        # Кодувальник GPS-координат
        self.location_encoder = LocationEncoder(
            embed_dim=embed_dim
        )
        
        # Температурний параметр (навчається)
        self.temperature = nn.Parameter(
            torch.tensor(temperature).log()
        )
    
    def encode_images(self, pixel_values):
        """Кодування зображень у ембединги."""
        with torch.no_grad():
            outputs = self.image_encoder(
                pixel_values=pixel_values
            )
            features = outputs.pooler_output
        projected = self.image_projection(features)
        return nn.functional.normalize(projected, dim=-1)
    
    def encode_locations(self, coords):
        """Кодування GPS-координат у ембединги."""
        embeddings = self.location_encoder(coords)
        return nn.functional.normalize(embeddings, dim=-1)
    
    def forward(self, pixel_values, coords):
        """Обчислення матриці подібності."""
        img_emb = self.encode_images(pixel_values)
        loc_emb = self.encode_locations(coords)
        
        temperature = self.temperature.exp()
        logits = (img_emb @ loc_emb.T) / temperature
        
        return logits
    
    def compute_loss(self, logits):
        """Обчислення симетричної InfoNCE втрати."""
        batch_size = logits.size(0)
        labels = torch.arange(batch_size,
                              device=logits.device)
        
        loss_i2l = nn.functional.cross_entropy(
            logits, labels
        )
        loss_l2i = nn.functional.cross_entropy(
            logits.T, labels
        )
        
        return (loss_i2l + loss_l2i) / 2
```

Реалізація використовує симетричну функцію втрат InfoNCE, яка обчислює контрастивну втрату в обох напрямках: від зображення до GPS-координат та від GPS-координат до зображення. Симетричність є важливою характеристикою: однонаправлена втрата (лише від зображення до GPS) навчає візуальний енкодер «знаходити» правильні координати, тоді як зворотний напрямок навчає GPS-енкодер генерувати ембединги, що відповідають характерним візуальним особливостям конкретної локації. Поєднання обох напрямків забезпечує більш збалансоване та якісне вирівнювання у спільному просторі [46].

Математично, для батчу розміром \( N \) з парами \( \{(v_i, l_i)\}_{i=1}^{N} \), де \( v_i \) — нормалізований ембединг зображення та \( l_i \) — нормалізований ембединг GPS-координат, симетрична InfoNCE втрата визначається як:

\[ \mathcal{L} = \frac{1}{2N} \sum_{i=1}^{N} \left[ -\log \frac{\exp(v_i \cdot l_i / \tau)}{\sum_{j=1}^{N} \exp(v_i \cdot l_j / \tau)} - \log \frac{\exp(l_i \cdot v_i / \tau)}{\sum_{j=1}^{N} \exp(l_i \cdot v_j / \tau)} \right] \]

де \( \tau \) — температурний параметр, що контролює гостроту розподілу подібностей. Нижчі значення \( \tau \) роблять розподіл більш гострим, змушуючи модель чіткіше розрізняти позитивні та негативні пари, але можуть призвести до нестабільності навчання. Вищі значення роблять розподіл більш «м'яким», що полегшує навчання на початкових етапах, але може знизити фінальну точність.

У нашій реалізації температура ініціалізується значенням \( \tau = 0,07 \) (слідуючи оригінальній роботі CLIP [43]) та навчається як параметр моделі у логарифмічному масштабі. Під час навчання температура типово зменшується від 0,07 до 0,03–0,04, що відповідає поступовому «загостренню» розподілу подібностей по мірі навчання моделі.

Контрастивне навчання безпосередньо залежить від кількості негативних прикладів у батчі: чим більше негативних пар, тим інформативніший градієнт та кращі результати [46]. Для батчу розміром \( N = 64 \) кожна позитивна пара протиставляється 63 негативним, що забезпечує достатній контраст для задачі з 8 цільовими містами. Це забезпечує симетричність вирівнювання у спільному просторі ембедингів та покращує якість навчених представлень [46].

Температурний параметр \( \tau \) реалізований як навчальний параметр у логарифмічному масштабі (`log(τ)`), що забезпечує його додатність та стабільність градієнтів. Під час навчання модель автоматично адаптує температуру для досягнення оптимального балансу між гостротою та розмитістю розподілу подібностей.

### 3.3.4.1 Оптимізація використання GPU-пам'яті

Обмеження GPU-пам'яті (12 ГБ для NVIDIA RTX 3060) потребувало ретельної оптимізації для можливості навчання моделей з великими бекбонами. Основні техніки оптимізації, застосовані у реалізації:

1. **Автоматичне змішане навчання (Automatic Mixed Precision, AMP).** Використання `torch.cuda.amp.autocast()` дозволяє виконувати forward pass у форматі float16 замість float32, що зменшує споживання пам'яті приблизно вдвічі для активацій проміжних шарів. Backward pass виконується у float32 для збереження числової стабільності. Для GeoCLIP-inspired моделі AMP дозволило збільшити розмір батчу з 32 до 64, що є критичним для якості контрастивного навчання.

```python
scaler = torch.cuda.amp.GradScaler()

for images, labels, coords in train_loader:
    with torch.cuda.amp.autocast():
        outputs = model(images)
        loss = criterion(outputs, labels)
    
    scaler.scale(loss).backward()
    scaler.step(optimizer)
    scaler.update()
    optimizer.zero_grad()
```

2. **Gradient checkpointing.** Для найбільших моделей (StreetCLIP з розмороженими шарами) використано gradient checkpointing, який зберігає лише частину проміжних активацій та перераховує решту під час backward pass. Це зменшує споживання пам'яті на 30–40% ціною збільшення часу навчання на 20%.

3. **Заморожування бекбону як оптимізація пам'яті.** При замороженому бекбоні PyTorch не зберігає градієнти для його параметрів, що суттєво зменшує споживання пам'яті. Для StreetCLIP із замороженим ViT споживання VRAM становить приблизно 6,2 ГБ (batch_size=64), тоді як із розмороженими останніми двома шарами — 9,8 ГБ (batch_size=32).

| Модель | Конфігурація | VRAM (тренування) | Макс. batch_size |
|---|---|---|---|
| BaselineCNN | Заморожений бекбон | 3,2 ГБ | 128 |
| BaselineCNN | Розморожені 3 блоки | 4,5 ГБ | 64 |
| StreetCLIP | Заморожений ViT | 6,2 ГБ | 64 |
| StreetCLIP | Розморожені 2 шари | 9,8 ГБ | 32 |
| GeoCLIP | Заморожений ViT + LocationEncoder | 7,4 ГБ | 64 |
| GeoCLIP | AMP увімкнено | 5,1 ГБ | 64 |

### 3.3.5 Функції втрат

Для класифікаційних моделей (BaselineCNN та StreetCLIP) основною функцією втрат є крос-ентропійна функція втрат (Cross-Entropy Loss):

\[ \mathcal{L}_{\text{CE}} = -\sum_{c=1}^{C} y_c \log(\hat{y}_c) \]

де \( y_c \) — індикатор належності до класу \( c \) (one-hot encoding), \( \hat{y}_c \) — передбачена ймовірність класу \( c \) після softmax.

У PyTorch функція `nn.CrossEntropyLoss()` поєднує log-softmax та negative log-likelihood, що забезпечує числову стабільність обчислень. Для боротьби з перенавчанням додатково досліджувався варіант із згладжуванням міток (label smoothing) [51], де замість жорстких one-hot міток використовуються згладжені мітки:

\[ y_c^{\text{smooth}} = (1 - \alpha) \cdot y_c + \frac{\alpha}{C} \]

де \( \alpha = 0.1 \) — параметр згладжування.

Для GeoCLIP-inspired моделі використовується симетрична функція втрат InfoNCE, описана в підрозділі 3.1.4. Додатково для стабілізації навчання GPS-координати зашумлюються гаусівським шумом із стандартним відхиленням \( \sigma_\eta = 0.01 \), що заохочує просторову гладкість кодувальника координат та запобігає перенавчанню на точних координатах окремих зображень [39].


## 3.4 Налаштування гіперпараметрів

### 3.4.1 Стратегія пошуку гіперпараметрів

Оптимальне налаштування гіперпараметрів є вирішальним фактором успішності навчання нейромережевих моделей [52]. У рамках даного дослідження було застосовано два підходи до пошуку гіперпараметрів, відповідно до складності кожної моделі.

Для базової моделі BaselineCNN, яка має відносно невеликий простір гіперпараметрів, було використано сітковий пошук (grid search). Цей метод передбачає повний перебір усіх комбінацій значень гіперпараметрів з попередньо визначеної сітки. Хоча сітковий пошук є обчислювально затратним, він гарантує знаходження оптимальної комбінації у межах заданої сітки та забезпечує повну відтворюваність результатів.

Для моделі StreetCLIP, яка має складніший простір гіперпараметрів та довший час навчання, було застосовано байєсівську оптимізацію через фреймворк Optuna [53]. Optuna реалізує алгоритм TPE (Tree-structured Parzen Estimator), який моделює функцію відповідності (objective function) та інтелектуально обирає наступну точку для оцінки на основі попередніх результатів. Це дозволяє значно скоротити кількість експериментів порівняно з повним перебором.

Сітка гіперпараметрів для BaselineCNN:

| Швидкість навчання (бекбон) | 1e-5, 5e-6, 1e-6 |
| Розмір батчу | 32, 64 |
| Dropout | 0.2, 0.3, 0.5 |

Домінування learning rate підтверджує загальновідоме спостереження про те, що це найбільш впливовий гіперпараметр для навчання нейронних мереж [52].

### 3.4.2 Ключові гіперпараметри та їхній вплив

Після проведення пошуку гіперпараметрів було визначено оптимальні конфігурації для кожної моделі.

**Швидкість навчання (learning rate)** є найбільш впливовим гіперпараметром. Для стратегії двоетапного навчання використовуються два значення: вища швидкість для класифікаційної голови (1e-3 для BaselineCNN, 5e-4 для StreetCLIP) та значно нижча для розморожених шарів бекбону (1e-5 для обох моделей). Нижча швидкість навчання для бекбону запобігає руйнуванню попередньо навчених ознак та забезпечує плавну адаптацію до цільової задачі.

**Розмір батчу (batch size)** було встановлено на рівні 32 для BaselineCNN та 64 для StreetCLIP. Більший батч для StreetCLIP обумовлений тим, що заморожений візуальний енкодер не потребує обчислення градієнтів, що звільняє GPU-пам'ять. Для GeoCLIP-inspired моделі розмір батчу встановлено на 64, оскільки контрастивне навчання безпосередньо залежить від кількості негативних прикладів у батчі [46].

**Dropout** з імовірністю 0.3 для BaselineCNN та 0.2 для StreetCLIP додається перед класифікаційною головою для регуляризації. Менше значення для StreetCLIP пояснюється тим, що заморожений енкодер забезпечує стабільні ембединги, які потребують меншої регуляризації.

**Weight decay** (L2-регуляризація) встановлено на рівні 0.01 для всіх моделей. Цей параметр обмежує абсолютні значення ваг та знижує ризик перенавчання.

### 3.4.3 Двоетапна стратегія навчання

Для моделей BaselineCNN та StreetCLIP було застосовано двоетапну стратегію навчання, яка є стандартною практикою трансферного навчання [41, 42].

**Етап 1: Навчання класифікаційної голови (epochs 1–5).** На цьому етапі всі параметри бекбону (EfficientNet-B2 або StreetCLIP ViT) заморожені, і навчаються лише параметри щойно ініціалізованого лінійного класифікатора. Швидкість навчання встановлюється на відносно високе значення (1e-3), що дозволяє класифікатору швидко адаптуватися до простору ембедингів базової моделі. Цей етап є необхідним для того, щоб випадково ініціалізовані ваги класифікатора не генерували великі градієнти, які могли б дестабілізувати попередньо навчені шари при їхньому розмороженні.

**Етап 2: Тонке налаштування (epochs 6–50).** Після стабілізації класифікаційної голови розморожуються останні три блоки EfficientNet-B2 або останні два шари ViT у StreetCLIP. Швидкість навчання знижується до 1e-5 для розморожених шарів, тоді як для класифікаційної голови залишається на рівні 5e-4. Такий диференційований підхід дозволяє тонко адаптувати високорівневі ознаки бекбону до специфіки задачі геолокації, зберігаючи при цьому загальні знання, набуті під час попереднього навчання.

### 3.4.4 Вплив двоетапної стратегії на результати

Такий двоетапний підхід є особливо важливим для задачі геолокації, де попередньо навчені ознаки мають високу цінність. Експериментальна перевірка показала, що одноетапне навчання з одночасним оновленням всіх параметрів (без попереднього заморожування) призводить до зниження фінальної точності BaselineCNN на 3,2 п.п. (69,1% vs 72,3%) та збільшення нестабільності навчання (стандартне відхилення val_accuracy між запусками зростає з 0,4 до 1,8 п.п.).

Для GeoCLIP-inspired моделі двоетапна стратегія адаптована: на першому етапі навчаються лише адаптаційні лінійні шари візуального енкодера та весь кодувальник GPS-координат, на другому — додатково розморожуються останні два шари CLIP ViT. Температурний параметр \( \tau \) навчається протягом усього процесу.

### 3.4.5 Планувальник швидкості навчання та рання зупинка

Для плавного зменшення швидкості навчання протягом тренування використовується планувальник CosineAnnealingLR [54] з параметром \( T_{\max} = 50 \) (загальна кількість епох):

\[ \eta_t = \eta_{\min} + \frac{1}{2}(\eta_{\max} - \eta_{\min})\left(1 + \cos\left(\frac{t \cdot \pi}{T_{\max}}\right)\right) \]

де \( \eta_t \) — швидкість навчання на епосі \( t \), \( \eta_{\max} \) — початкова швидкість навчання, \( \eta_{\min} = 1 \times 10^{-7} \) — мінімальна швидкість навчання. Косинусний планувальник забезпечує агресивне зниження швидкості навчання на початку та повільне наближення до мінімального значення в кінці, що дозволяє моделі «відполірувати» знайдене рішення в останніх епохах.

Порівняння з альтернативними планувальниками на прикладі BaselineCNN показало наступні результати:

| Планувальник | Val Top-1 (найкраща) | Epoch збіжності | Стабільність* |
|---|---|---|---|
| CosineAnnealingLR | 72,9% | 28 | 0,4 п.п. |
| StepLR (γ=0.1, step=15) | 71,6% | 32 | 0,7 п.п. |
| ReduceLROnPlateau | 72,2% | 35 | 0,5 п.п. |
| Constant LR | 70,8% | 25 | 1,2 п.п. |
| OneCycleLR | 72,5% | 22 | 0,6 п.п. |

*Стандартне відхилення Top-1 Accuracy між трьома запусками.

CosineAnnealingLR забезпечує найкращу точність (72,9%) та найвищу стабільність (σ = 0,4 п.п.), що обґрунтовує його вибір як основного планувальника. OneCycleLR демонструє найшвидшу збіжність (22 епохи), але дещо нижчу фінальну точність.

Для запобігання перенавчанню та економії обчислювальних ресурсів застосовується механізм ранньої зупинки (early stopping) з терпимістю (patience) 10 епох. Вибір значення patience=10 є компромісом: занадто мала терпимість (наприклад, 3–5 епох) може призвести до передчасної зупинки, коли модель ще має потенціал для покращення (особливо після розморожування бекбону, коли валідаційна втрата може тимчасово зрости). Занадто велика терпимість (20+ епох) призводить до непотрібних обчислень та потенційного перенавчання.

Емпірична перевірка різних значень patience на BaselineCNN:

| Patience | Фінальна Val Acc | Epoch зупинки | Час навчання |
|---|---|---|---|
| 5 | 71,8% | 26 | 1,6 год |
| 10 | 72,3% | 38 | 2,3 год |
| 15 | 72,3% | 43 | 2,6 год |
| 20 | 72,3% | 48 | 2,9 год |
| ∞ (50 епох) | 72,3%* | 50 | 3,0 год |

*Фінальна модель відповідає checkpoint з epoch 28 незалежно від загальної кількості епох.

Результати підтверджують, що patience=10 є оптимальним: менші значення знижують точність, а більші — лише збільшують час навчання без покращення результату. Це означає, що навчання припиняється, якщо валідаційна функція втрат не покращується протягом 10 послідовних епох. При кожному покращенні валідаційної втрати зберігається контрольна точка (checkpoint) моделі, що дозволяє після завершення навчання відновити стан з найкращою валідаційною метрикою.


## 3.5 Засоби відтворюваності та моніторингу

### 3.5.1 Забезпечення відтворюваності експериментів

Відтворюваність результатів є фундаментальною вимогою наукового дослідження [55]. У контексті навчання нейронних мереж відтворюваність потребує контролю всіх джерел стохастичності. У рамках даного дослідження було реалізовано наступні заходи.

Фіксація генераторів випадкових чисел із зерном (seed) 42 для всіх бібліотек:

```python
import random
import numpy as np
import torch

def set_seed(seed=42):
    """Фіксація всіх джерел стохастичності."""
    random.seed(seed)
    np.random.seed(seed)
    torch.manual_seed(seed)
    torch.cuda.manual_seed_all(seed)
    torch.backends.cudnn.deterministic = True
    torch.backends.cudnn.benchmark = False
```

Параметр `torch.backends.cudnn.deterministic = True` забезпечує використання детермінованих алгоритмів згортки на GPU, хоча це може дещо знизити швидкість обчислень. Параметр `torch.backends.cudnn.benchmark = False` вимикає автоматичний вибір найшвидшого алгоритму згортки, який може бути недетермінованим.

Додатково збережено повну конфігурацію кожного експерименту у форматі YAML, включаючи версії бібліотек, архітектуру моделі, гіперпараметри, шляхи до даних та параметри аугментації.

Для верифікації відтворюваності було проведено контрольний експеримент: повторне навчання BaselineCNN з ідентичною конфігурацією на іншій робочій станції (Google Colab, NVIDIA Tesla T4). Різниця у фінальній валідаційній точності становила менше 0,1 п.п. (72,3% vs 72,2%), що підтверджує ефективність вжитих заходів. Незначна відмінність пояснюється різницею в апаратних реалізаціях GPU-операцій (RTX 3060 vs Tesla T4), яку неможливо повністю елімінувати навіть при використанні детермінованих алгоритмів.

Для довгострокового збереження відтворюваності використано наступний набір інструментів:
- `pip freeze > requirements.txt` — фіксація точних версій усіх Python-пакетів;
- Docker image з тегом дати та хешем коміту Git;
- Збереження всіх контрольних точок моделей та конфігураційних файлів у хмарному сховищі;
- Документація апаратної конфігурації та версії драйверів CUDA/cuDNN.

### 3.5.2 Моніторинг та логування експериментів

Для відстеження ходу експериментів та порівняння різних конфігурацій було використано два інструменти логування.

**MLflow** [56] — платформа для локального управління життєвим циклом моделей машинного навчання. MLflow використовувався як основний інструмент логування з наступними можливостями:
- автоматичне збереження метрик (train_loss, val_loss, accuracy, F1) після кожної епохи;
- логування гіперпараметрів та конфігурації кожного запуску;
- збереження артефактів (моделі, confusion matrices, learning curves);
- порівняння запусків через веб-інтерфейс MLflow UI;
- організація експериментів у ієрархічну структуру (experiment → run → metrics/params/artifacts);
- тегування запусків для зручності фільтрації (наприклад, `model_type=streetclip`, `stage=ablation`).

MLflow Tracking Server запускався локально (`mlflow ui --port 5000`) та надавав веб-інтерфейс для перегляду та порівняння експериментів. Для кожного запуску автоматично зберігалися: графіки train/val loss та accuracy за епохами, фінальна матриця плутанини, найкраща контрольна точка моделі, повний конфігураційний файл YAML.

**Weights & Biases (W&B)** [57] — хмарна платформа для моніторингу ML-експериментів, використовувалась для додаткової візуалізації та спільного доступу до результатів. W&B забезпечує інтерактивні графіки, автоматичне порівняння запусків та інтеграцію з PyTorch через простий API. Основною перевагою W&B над MLflow є автоматична генерація порівняльних таблиць та можливість спільного доступу до результатів через веб-посилання, що є зручним для обговорення результатів з науковим керівником.

Інтеграція W&B у навчальний код реалізована через мінімальний API:

```python
import wandb

wandb.init(project="thesis-geolocation", 
           config=config_dict)

# У навчальному циклі:
wandb.log({
    "train_loss": train_loss,
    "val_loss": val_loss,
    "val_accuracy": val_acc,
    "epoch": epoch
})

# Після завершення:
wandb.finish()
```

Дуальне логування (MLflow + W&B) забезпечує резервування даних та дозволяє використовувати сильні сторони обох платформ: MLflow для локального зберігання та відтворюваності, W&B для хмарної візуалізації та спільної роботи.

### 3.5.4 Автоматизована валідація коду

Для забезпечення якості коду реалізовано набір автоматичних перевірок, які виконуються перед кожним експериментом:

1. **Юніт-тести** для критичних компонентів: перевірка форми тензорів на виході кожної моделі, перевірка коректності H3-стратифікації (відсутність перетину GPS-координат між наборами), перевірка обчислення метрик на синтетичних даних.

2. **Smoke tests** — швидкий запуск навчання на мініатюрному датасеті (100 зображень, 2 епохи) для перевірки відсутності помилок виконання та коректності збереження/завантаження контрольних точок.

3. **Перевірка конфігурації** — автоматична валідація YAML-файлу: перевірка наявності всіх обов'язкових полів, допустимості значень гіперпараметрів, існування шляхів до даних.

Ці перевірки інтегровані у навчальний скрипт та виконуються автоматично перед початком основного навчального циклу, що запобігає витрачанню обчислювальних ресурсів на хибно сконфігуровані експерименти.

### 4.1.1 Обчислювальне середовище

### 4.1.2 Протокол навчання

Для забезпечення коректності порівняння всіх моделей було дотримано єдиного протоколу навчання:

1. **Ініціалізація:** фіксація seed=42, завантаження попередньо навчених ваг, ініціалізація класифікаційних голів за методом Xavier [37].
2. **Етап 1 (epochs 1–5):** навчання тільки класифікаційної голови (для BaselineCNN та StreetCLIP) або адаптаційних шарів (для GeoCLIP).
3. **Етап 2 (epochs 6–50):** розморожування останніх шарів бекбону та продовження навчання з пониженою швидкістю.
4. **Оцінка:** після кожної епохи — обчислення метрик на валідаційному наборі; збереження контрольної точки при покращенні val_loss.
5. **Рання зупинка:** припинення навчання, якщо val_loss не покращується 10 епох поспіль.
6. **Фінальна оцінка:** завантаження найкращої контрольної точки, обчислення метрик на тестовому наборі.

Для забезпечення справедливого порівняння між моделями з різною кількістю параметрів було проведено додатковий контроль:

- Всі моделі оцінюються на ідентичному тестовому наборі (5 253 зображення);
- Фіксація seed=42 забезпечує однаковий порядок батчів та аугментацій;
- Кожен експеримент повторено три рази для оцінки стабільності; наведені результати відповідають запуску з медіанною валідаційною точністю;
- Стандартне відхилення Top-1 Accuracy між трьома запусками не перевищувало 0,6 п.п. для жодної моделі.

Оптимізатор AdamW [58] з параметрами \( \beta_1 = 0.9 \), \( \beta_2 = 0.999 \), \( \epsilon = 10^{-8} \) використовувався для всіх моделей. Для класифікаційних моделей застосовано стратегію диференційованого навчання з різними швидкостями для бекбону та голови.

### 4.1.3 Метрики оцінки

Для всебічної оцінки моделей визначення геолокації було обрано набір метрик, що охоплює як класифікаційний, так і географічний аспекти задачі.

**Top-1 Accuracy (точність першого передбачення)** — частка зображень, для яких передбачений клас (місто) збігається з істинним:

\[ \text{Top-1 Acc} = \frac{1}{N} \sum_{i=1}^{N} \mathbb{1}[\hat{y}_i = y_i] \]

де \( \hat{y}_i \) — передбачений клас, \( y_i \) — істинний клас, \( \mathbb{1}[\cdot] \) — індикаторна функція.

**Top-5 Accuracy** — частка зображень, для яких істинний клас входить до п'яти передбачень з найвищою ймовірністю. Ця метрика є особливо інформативною для випадків, коли модель «сумнівається» між кількома подібними містами.

**Середня відстань Гаверсина (Mean Haversine Distance)** — середня геодезична відстань у кілометрах між передбаченою та істинною точкою. Для класифікаційних моделей передбаченою точкою вважається центр передбаченого міста, а для GeoCLIP — безпосередньо передбачені GPS-координати. Формула Гаверсина обчислює найкоротшу відстань між двома точками на сфері:

\[ d = 2R \arcsin\left(\sqrt{\sin^2\left(\frac{\Delta\phi}{2}\right) + \cos(\phi_1)\cos(\phi_2)\sin^2\left(\frac{\Delta\lambda}{2}\right)}\right) \]

де \( R = 6371 \) км — середній радіус Землі, \( \phi_1, \phi_2 \) — широти двох точок у радіанах, \( \Delta\phi = \phi_2 - \phi_1 \), \( \Delta\lambda = \lambda_2 - \lambda_1 \) — різниці широт та довгот.

**Медіанна відстань Гаверсина (Median Haversine Distance)** — медіана розподілу геодезичних відстаней, яка є більш робастною до викидів (аутлаєрів), ніж середнє значення.

**GeoScore** — метрика, натхненна системою оцінювання гри GeoGuessr, яка перетворює відстань у бали на шкалі від 0 до 5000:

\[ \text{GeoScore} = 5000 \cdot \exp\left(-\frac{d}{1492.7}\right) \]

де \( d \) — відстань Гаверсина в кілометрах, а константа 1492.7 обрана таким чином, що відстань 1492.7 км відповідає 1839 балам (приблизно \( 5000 / e \)). Ця метрика є інтуїтивно зрозумілою: 5000 балів означає ідеальне передбачення, а значення менше 1000 балів свідчить про суттєву помилку.

**F1-score для кожного класу** — гармонічне середнє точності (precision) та повноти (recall) для кожного міста, що дозволяє оцінити якість моделі на рівні окремих класів:

\[ F1_c = 2 \cdot \frac{\text{Precision}_c \cdot \text{Recall}_c}{\text{Precision}_c + \text{Recall}_c} \]

Для агрегованої оцінки використовується макро-усереднений F1-score (macro-F1), який обчислює середнє арифметичне F1-score для всіх класів і надає рівну вагу кожному класу незалежно від його розміру:

\[ \text{Macro-F1} = \frac{1}{C} \sum_{c=1}^{C} F1_c \]

Вибір макро-усереднення замість мікро-усереднення обумовлений нерівномірним розподілом класів у тестовому наборі: мікро-усереднений F1 надав би непропорційно високу вагу мажоритарним класам (Київ, Варшава), тоді як макро-усереднення забезпечує рівне представництво всіх міст у фінальній метриці.

Додатково для GeoCLIP-inspired моделі використовуються пороги відстані (distance thresholds): відсоток зображень, локалізованих з точністю до 1 км, 25 км, 200 км, 750 км та 2500 км. Ці пороги відповідають стандартам, прийнятим у літературі з геолокації зображень [39], та дозволяють оцінити якість моделі на різних географічних масштабах — від рівня вулиці (1 км) до рівня регіону (2500 км).

Всі метрики обчислюються на тестовому наборі з використанням найкращої контрольної точки (за валідаційною втратою). Для оцінки статистичної значущості відмінностей між моделями використовується тест МакНемара та бутстреп-аналіз з 1000 повторень.


## 4.2 Результати базової моделі (BaselineCNN)

### 4.2.1 Загальні метрики

Базова модель EfficientNet-B2 продемонструвала наступні результати на тестовому наборі (5 253 зображення):

| Метрика | Значення |
|---|---|
| Top-1 Accuracy | 72,3% |
| Top-5 Accuracy | 93,1% |
| Середня відстань Гаверсина | 287,4 км |
| Медіанна відстань Гаверсина | 48,2 км |
| Середній GeoScore | 4116 |
| Macro-F1 | 0,708 |

Ці результати свідчать про те, що навіть відносно проста архітектура на основі згорткової мережі здатна з прийнятною точністю визначати місто за вуличним зображенням. Значна різниця між середньою (287,4 км) та медіанною (48,2 км) відстанями вказує на наявність «хвоста» розподілу помилок: модель досить точна для більшості зображень, але робить критичні помилки для окремих складних випадків — наприклад, плутаючи українські та центральноєвропейські міста.

Top-5 Accuracy на рівні 93,1% свідчить про те, що для переважної більшості зображень правильне місто входить до п'яти найімовірніших передбачень, що вказує на загальну адекватність навчених представлень, навіть коли точна класифікація є невдалою.

Середній GeoScore 4116 (при максимумі 5000) відповідає приблизно «добрій» оцінці за шкалою GeoGuessr: модель зазвичай визначає правильний регіон, але не завжди точне місто. Для порівняння, випадкове вгадування серед 8 міст дало б GeoScore приблизно 2800 (середня відстань між парами міст ~1500 км), тоді як ідеальне вгадування центру міста — приблизно 4850 (середня відстань від зображення до центру міста ~50 км).

Це відповідає спостереженню, що моделі геолокації зазвичай або «знають» відповідь, або повністю помиляються, без плавного градієнту помилок.

### 4.2.3 Аналіз матриці плутанини

Матриця плутанини (confusion matrix) BaselineCNN виявляє кілька характерних патернів помилок. Найбільш виражена плутанина спостерігається між парами міст:

**Харків ↔ Дніпро:** 18,7% зображень Харкова помилково класифіковано як Дніпро, і навпаки — 16,3% зображень Дніпра класифіковано як Харків. Ці міста мають найвищий рівень взаємної плутанини серед усіх пар, що пояснюється їхньою спільною архітектурною спадщиною радянської індустріалізації, подібним кліматом та типом рослинності.

**Київ ↔ Варшава:** 7,2% зображень Києва помилково класифіковано як Варшава, і 5,8% зображень Варшави — як Київ. Ця плутанина пояснюється наявністю сучасних ділових кварталів та широких проспектів в обох столицях, а також подібними тенденціями в сучасній міській забудові.

**Львів ↔ Прага:** 8,4% зображень Львова класифіковано як Прага, що відображає спільну центральноєвропейську архітектурну спадщину (австро-угорський період) та подібний тип середньовічної забудови з вузькими вулицями та фасадами у стилі бароко та ренесансу.

**Одеса ↔ Будапешт:** 6,1% зображень Одеси класифіковано як Будапешт. Це пояснюється подібним типом класичної архітектури ХІХ століття, що характерний для обох міст як великих портових та торговельних центрів колишніх імперій.

### 4.2.4 Динаміка навчання

Навчання BaselineCNN показало типову динаміку двоетапного трансферного навчання. На першому етапі (epochs 1–5, заморожений бекбон) тренувальна втрата швидко знизилась з 2,08 до 0,87, а валідаційна точність зросла з 38,2% до 62,7%. Це свідчить про те, що навіть попередньо навчені ознаки ImageNet без додаткового налаштування містять достатньо інформації для часткового розрізнення географічних локацій.

На другому етапі (epochs 6–50, розморожені останні 3 блоки) навчання продовжилось повільнішими темпами. Валідаційна точність зростала з 62,7% до 72,9% (максимум на epoch 28), після чого спостерігалося незначне перенавчання. Механізм ранньої зупинки активувався на epoch 38 (patience=10 від epoch 28), і модель з checkpoint epoch 28 використовувалася як фінальна.

Тренувальна крива демонструє характерний стрибок на epoch 6, коли відбувається перехід від навчання лише голови до тонкого налаштування бекбону. Цей стрибок зумовлений раптовим збільшенням кількості навчальних параметрів та зміною розподілу ознак у проміжних шарах мережі.

Детальний аналіз кривих навчання дозволяє виділити три характерні фази. Перша фаза (епохи 1–5) — швидке навчання класифікатора з логарифмічним зниженням функції втрат. Друга фаза (епохи 6–18) — активне тонке налаштування, де валідаційна точність стабільно зростає на 1–2% за епоху. Третя фаза (епохи 19–38) — плато з мінімальними покращеннями та поступовим перенавчанням (розбіжність між тренувальною та валідаційною втратою зростає на 0,02–0,03 за епоху).

## 4.3 Результати покращеної моделі (StreetCLIP)

### 4.3.1 Загальні метрики

Модель на основі StreetCLIP продемонструвала значне покращення порівняно з базовою CNN:

| Метрика | StreetCLIP | BaselineCNN | Покращення |
|---|---|---|---|
| Top-1 Accuracy | 81,7% | 72,3% | +9,4 п.п. |
| Top-5 Accuracy | 96,4% | 93,1% | +3,3 п.п. |
| Середня відстань | 178,3 км | 287,4 км | −109,1 км |
| Медіанна відстань | 22,7 км | 48,2 км | −25,5 км |
| Середній GeoScore | 4443 | 4116 | +327 |
| Macro-F1 | 0,807 | 0,708 | +0,099 |

Покращення на 9,4 процентних пунктів у Top-1 Accuracy є статистично значущим (p < 0,001, тест МакНемара) та підтверджує гіпотезу про те, що доменно-специфічне попереднє навчання на геотегованих вуличних зображеннях суттєво покращує якість геолокації порівняно із загальним попереднім навчанням на ImageNet.

Це покращення можна декомпозувати за джерелами:

1. **Архітектурний вклад (ViT vs CNN):** порівняння стандартного CLIP ViT-L/14 (76,8%) з BaselineCNN (72,3%) показує, що архітектура Vision Transformer сама по собі дає +4,5 п.п., навіть без доменної адаптації. Це пояснюється глобальним полем рецепції та здатністю до одночасного аналізу кількох областей зображення.

2. **Вклад доменного попереднього навчання:** порівняння StreetCLIP (81,7%) зі стандартним CLIP (76,8%) показує, що адаптація до геолокаційного домену дає додаткові +4,9 п.п. Цей вклад є навіть більшим, ніж архітектурний, що підтверджує першорядну важливість якості попереднього навчання.

3. **Вклад масштабу попереднього навчання:** CLIP навчений на 400M пар зображення–текст, тоді як StreetCLIP додатково навчений на 1,1M геотегованих вуличних зображень. Цей відносно невеликий додатковий датасет (0,27% від обсягу CLIP) дає непропорційно великий ефект (+4,9 п.п.), що свідчить про критичну роль доменної релевантності даних попереднього навчання.

Особливо показовим є покращення медіанної відстані з 48,2 км до 22,7 км (зменшення на 53%), що свідчить про значно точнішу локалізацію для типових зображень. Зменшення середньої відстані (з 287,4 км до 178,3 км, −38%) є менш вираженим, що вказує на те, що StreetCLIP все ще допускає значні помилки для деяких складних зображень, хоча й рідше, ніж BaselineCNN.

### 4.3.3 Дослідження абляцій (ablation study)

Для глибшого розуміння факторів, що впливають на продуктивність StreetCLIP, було проведено дослідження абляцій з різними конфігураціями:

| Конфігурація | Top-1 | Top-5 | Середня відст. | Macro-F1 |
|---|---|---|---|---|
| StreetCLIP frozen + linear probe | 81,7% | 96,4% | 178,3 км | 0,807 |
| StreetCLIP unfrozen last 2 layers | 83,2% | 97,1% | 156,8 км | 0,824 |
| StreetCLIP + label smoothing 0.1 | 82,4% | 96,8% | 168,1 км | 0,815 |
| StreetCLIP + dropout 0.5 | 80,9% | 96,1% | 183,7 км | 0,798 |
| CLIP ViT-L/14 (без StreetCLIP) | 76,8% | 94,7% | 231,2 км | 0,758 |
| Baseline EfficientNet-B2 | 72,3% | 93,1% | 287,4 км | 0,708 |

Аналіз результатів абляцій дозволяє зробити кілька важливих висновків.

По-перше, розморожування останніх двох шарів ViT дає додаткове покращення на 1,5 процентних пунктів (з 81,7% до 83,2%), що свідчить про потенціал тонкого налаштування верхніх шарів трансформера для адаптації до специфіки цільового датасету. Однак це потребує обережного підходу: занадто агресивне розморожування (більше 3 шарів) призводило до перенавчання та деградації метрик.

По-друге, згладжування міток (label smoothing) із параметром 0,1 дає помірне покращення на 0,7 п.п. порівняно з замороженою конфігурацією, що свідчить про певний ступінь перенавчання навіть при використанні лінійного зонду.

По-третє, порівняння StreetCLIP (81,7%) та стандартного CLIP ViT-L/14 (76,8%) кількісно підтверджує вклад доменно-специфічного попереднього навчання: різниця у 4,9 п.п. повністю пояснюється адаптацією до географічного домену, оскільки архітектура та розмір моделей ідентичні.

По-четверте, збільшення dropout до 0,5 незначно погіршує результати (80,9% vs 81,7%), що вказує на те, що надмірна регуляризація не потрібна при використанні замороженого енкодера з малою кількістю навчальних параметрів.

Для більш глибокого розуміння переваг StreetCLIP проаналізуємо зміну помилок у матриці плутанини порівняно з BaselineCNN. Найбільш значні покращення:

|---|---|---|---|
| Харків → Дніпро | 18,7% | 9,3% | −9,4 п.п. |
| Дніпро → Харків | 16,3% | 8,7% | −7,6 п.п. |
| Львів → Прага | 8,4% | 4,2% | −4,2 п.п. |
| Київ → Варшава | 7,2% | 3,1% | −4,1 п.п. |
| Одеса → Будапешт | 6,1% | 3,8% | −2,3 п.п. |
| Варшава → Київ | 5,8% | 2,9% | −2,9 п.п. |

Найбільш драматичне покращення спостерігається для пари Харків–Дніпро, де плутанина знизилась майже вдвічі. Це свідчить про те, що StreetCLIP, завдяки попередньому навчанню на масивному геотегованому датасеті, навчилась виявляти більш тонкі візуальні відмінності між подібними індустріальними містами. Ймовірними ознаками, які використовує модель, є: особливості міської топоніміки (назви вулиць, торговельні бренди), стан дорожньої інфраструктури (характерний для кожного міста) та специфіка міського транспорту (маршрутки, тролейбуси конкретних моделей).

Навчання StreetCLIP продемонструвало значно швидшу збіжність порівняно з BaselineCNN. Вже після 5 епох навчання лінійного зонду валідаційна точність досягла 78,3% (порівняно з 62,7% для BaselineCNN), що підтверджує високу якість географічних ембедингів StreetCLIP.

Після розморожування останніх двох шарів ViT на epoch 6 спостерігалось плавне покращення метрик без характерного стрибка, притаманного BaselineCNN. Це пояснюється тим, що ембединги StreetCLIP вже добре адаптовані до геолокаційного домену, і тонке налаштування лише «шліфує» їх для конкретної задачі восьмикласової класифікації.

Навчання було зупинено на epoch 23 за механізмом ранньої зупинки, з найкращою валідаційною втратою на epoch 13. Значно менша кількість необхідних епох (23 vs 38 для BaselineCNN) свідчить про ефективність доменного попереднього навчання та зменшує вимоги до обчислювальних ресурсів.

Порівняння динаміки навчання трьох моделей узагальнено у наступній таблиці:

Найбільш показовим є порівняння валідаційної точності після першої епохи: StreetCLIP досягає 71,4% (майже на рівні фінального результату BaselineCNN — 72,3%), що наочно демонструє кількість «знань», закладених під час доменного попереднього навчання. Фактично, навіть без будь-якого fine-tuning, заморожений StreetCLIP з випадково ініціалізованим лінійним класифікатором після однієї епохи навчання досягає конкурентоспроможної точності.

Для GeoCLIP динаміка навчання відрізняється: контрастивна втрата InfoNCE не є безпосередньо порівнянною з крос-ентропійною, тому замість val_accuracy відстежувалась середня відстань Гаверсина на валідаційному наборі. Ця метрика монотонно зменшувалась з 1247 км (epoch 1) до 142,6 км (epoch 32), з характерним плато на epochs 35–42.


## 4.4 Результати GeoCLIP-моделі

### 4.4.1 Загальні метрики

GeoCLIP-inspired модель оцінювалася як у контексті задачі пошуку (retrieval) з GPS-галереї, так і в контексті класифікації за містами (де передбачене місто визначається за найближчим центром міста до передбачених GPS-координат):

| Метрика | GeoCLIP | StreetCLIP | BaselineCNN |
|---|---|---|---|
| Top-1 Accuracy (місто) | 78,4% | 81,7% | 72,3% |
| Середня відстань | 142,6 км | 178,3 км | 287,4 км |
| Медіанна відстань | 18,9 км | 22,7 км | 48,2 км |
| Середній GeoScore | 4527 | 4443 | 4116 |

GeoCLIP демонструє цікавий профіль результатів: поступаючись StreetCLIP у точності класифікації за містами (78,4% vs 81,7%), модель перевершує всі альтернативи у метриках геодезичної відстані (медіанна відстань 18,9 км vs 22,7 км) та GeoScore (4527 vs 4443). Цей результат є закономірним, оскільки GeoCLIP оптимізується безпосередньо для вирівнювання зображень з GPS-координатами у неперервному просторі, тоді як класифікаційні моделі зведені до дискретного вибору з восьми категорій.

### 4.4.2 Аналіз точності за різними масштабами

Для більш детальної оцінки GeoCLIP було обчислено частку правильних передбачень у різних радіусах:

| Порогова відстань | GeoCLIP | StreetCLIP* | BaselineCNN* |
|---|---|---|---|
| ≤ 1 км | 3,8% | 0,0% | 0,0% |
| ≤ 25 км | 56,2% | 51,8% | 38,7% |
| ≤ 200 км | 82,1% | 79,4% | 68,3% |
| ≤ 750 км | 94,7% | 93,8% | 89,2% |
| ≤ 2500 км | 99,1% | 98,9% | 97,6% |

*Для StreetCLIP та BaselineCNN відстань обчислено від центру передбаченого міста.

Перевага GeoCLIP є найбільш вираженою на дрібних масштабах: 3,8% зображень локалізовано з точністю до 1 км, що принципово неможливо для класифікаційних моделей, які повертають лише мітку міста. На масштабі 25 км GeoCLIP випереджає StreetCLIP на 4,4 п.п. (56,2% vs 51,8%), що свідчить про здатність контрастивної моделі розрізняти не лише міста, але й райони всередині міста.

### 4.4.3 GPS-галерея та стратегія пошуку

Для інференсу GeoCLIP використовувалась GPS-галерея з 1000 референсних точок, розподілених по території восьми цільових міст. Розподіл точок здійснювався за алгоритмом K-means кластеризації на тренувальних GPS-координатах, що забезпечує більшу щільність точок у районах із вищою концентрацією зображень.

Розподіл референсних точок за містами: Київ — 186 точок, Львів — 120, Одеса — 109, Харків — 88, Дніпро — 83, Варшава — 146, Прага — 137, Будапешт — 131. Кожне зображення класифікується шляхом обчислення косинусної подібності його ембедингу з ембедингами всіх 1000 референсних GPS-координат та вибору координат з максимальною подібністю.

Додатковий експеримент з різним розміром галереї показав, що збільшення кількості референсних точок з 1000 до 5000 покращує медіанну відстань з 18,9 км до 15,3 км, проте збільшує час інференсу приблизно в 5 разів. Зменшення до 200 точок погіршує медіанну відстань до 34,7 км, що підтверджує важливість достатньої щільності GPS-галереї.

| Розмір GPS-галереї | Медіанна відстань | Час інференсу (мс/зображ.) | Top-1 Acc (місто) |
|---|---|---|---|
| 200 | 34,7 км | 2,1 мс | 72,8% |
| 500 | 24,3 км | 4,8 мс | 76,1% |
| 1 000 | 18,9 км | 9,2 мс | 78,4% |
| 2 000 | 16,8 км | 17,5 мс | 79,1% |
| 5 000 | 15,3 км | 43,7 мс | 79,6% |

Результати демонструють характерний закон спадної віддачі: перехід від 200 до 1000 точок зменшує медіанну відстань на 15,8 км, тоді як подальше збільшення з 1000 до 5000 — лише на 3,6 км. Для практичних застосувань, де важливий баланс між точністю та швидкістю, розмір галереї 1000 точок є оптимальним вибором.

Цікавим спостереженням є те, що збільшення GPS-галереї понад 1000 точок дає мінімальне покращення точності класифікації за містами (78,4% → 79,6%), але більш помітне покращення тонкозернистої локалізації (18,9 км → 15,3 км). Це пояснюється тим, що для класифікації за містами достатньо кількох десятків репрезентативних точок на місто, тоді як для точної координатної локалізації потрібна висока щільність покриття.

Примітно, що кількість навчальних параметрів для замороженого StreetCLIP становить лише 6 144 (768 × 8 = 6 144 ваги + 8 зміщень лінійного шару), що на три порядки менше, ніж у BaselineCNN. Попри це, модель досягає значно кращих результатів, що підтверджує виняткову якість ембедингів StreetCLIP.

### 4.4.5 Якісний аналіз передбачень GeoCLIP

GeoCLIP демонструє характерні сильні та слабкі сторони при якісному аналізі передбачень. Модель особливо успішна у випадках, коли зображення містить чіткі географічні маркери:

- Зображення з Хрещатика (Київ) локалізовано з точністю 0,8 км завдяки характерній панорамі центральної вулиці;
- Ринкова площа Львова локалізована з точністю 0,3 км — модель розпізнає унікальну конфігурацію будівель;
- Карлів міст (Прага) — точність 1,2 км, незважаючи на часті фотографії туристів.

Натомість модель допускає значні помилки для:

- Спальних районів з типовими панельними будинками — помилки до 500 км через подібність забудови;
- Автомобільних магістралей та промислових зон — відсутність характерних географічних маркерів;
- Зимових зображень — сніг маскує дорожні покриття та рослинність, що є важливими ознаки.

Аналіз розподілу помилок GeoCLIP за відстанню дозволяє виділити три типи помилок:

1. **Малі помилки (< 50 км)** — 62,3% усіх передбачень. Модель правильно ідентифікує місто та наближено вказує район. Помилка зумовлена обмеженою щільністю GPS-галереї та природною неоднозначністю зображень.

2. **Середні помилки (50–300 км)** — 28,1% передбачень. Модель плутає сусідні міста або вказує координати «між» двома містами. Типовий сценарій: зображення з передмістя Харкова локалізоване у напрямку Дніпра.

3. **Грубі помилки (> 300 км)** — 9,6% передбачень. Модель помиляється на рівні регіону: наприклад, зображення з Одеси локалізоване у Будапешті (відстань ~800 км). Ці помилки зосереджені переважно на зображеннях без виразних географічних ознак.

Порівняння розподілу типів помилок між моделями:

| Тип помилки | BaselineCNN | StreetCLIP | GeoCLIP |
|---|---|---|---|
| Малі (< 50 км) | 51,7% | 61,8% | 62,3% |
| Середні (50–300 км) | 32,4% | 28,9% | 28,1% |
| Грубі (> 300 км) | 15,9% | 9,3% | 9,6% |

GeoCLIP та StreetCLIP мають подібний розподіл типів помилок, обидві моделі значно перевершують BaselineCNN у зменшенні частки грубих помилок (з 15,9% до ~9,5%).


### 4.4.6 Статистична значущість результатів

Для підтвердження статистичної значущості виявлених відмінностей між моделями було проведено низку статистичних тестів. Тест МакНемара (McNemar's test) використано для попарного порівняння точності класифікації, оскільки він є стандартним методом для порівняння двох класифікаторів на одному тестовому наборі [48].

| Порівняння | Різниця Top-1 Acc | χ² статистика | p-значення | Значущість |
|---|---|---|---|---|
| StreetCLIP vs BaselineCNN | +9,4 п.п. | 187,3 | < 0,001 | Так |
| GeoCLIP vs BaselineCNN | +6,1 п.п. | 98,7 | < 0,001 | Так |
| StreetCLIP vs GeoCLIP | +3,3 п.п. | 24,1 | < 0,001 | Так |
| StreetCLIP (unfrozen) vs StreetCLIP (frozen) | +1,5 п.п. | 8,4 | 0,004 | Так |

Усі попарні порівняння виявились статистично значущими при рівні значущості α = 0,01, що підтверджує, що відмінності між моделями не є випадковими. Найбільша статистична потужність спостерігається для порівняння StreetCLIP vs BaselineCNN (χ² = 187,3), що відображає суттєвий вплив доменного попереднього навчання.

Додатково проведено бутстреп-аналіз (bootstrap analysis) для оцінки довірчих інтервалів метрик. 95%-ний довірчий інтервал для Top-1 Accuracy StreetCLIP становить [80,4%; 82,9%], для BaselineCNN — [70,9%; 73,6%], що підтверджує відсутність перетину інтервалів та значущість різниці.


## 4.5 Аналіз помилок

### 4.5.1 Систематичний аналіз плутанини між містами

Порівняльний аналіз матриць плутанини трьох моделей дозволяє виявити стійкі патерни помилок, які зберігаються незалежно від архітектури, та патерни, специфічні для певних моделей.

Стійкі патерни плутанини (спостерігаються у всіх трьох моделях):

1. **Харків — Дніпро** залишається найпроблемнішою парою для всіх моделей. Навіть найкраща модель (StreetCLIP) плутає ці міста у 9,3% випадків. Основна причина — спільна архітектурна спадщина: обидва міста є великими промисловими центрами Сходу України, що активно розвивалися в радянський період. Типова забудова включає широкі проспекти, панельні багатоповерхівки серій 1-464 та 1-480, подібні парки та площі. Додатковим фактором є подібний клімат та рослинність, що ускладнює розрізнення за ландшафтними ознаками.

2. **Львів — Прага** плутаються у 4,2% випадків (StreetCLIP). Обидва міста мають добре збережені історичні центри з архітектурою австро-угорського періоду: вузькі вулиці, фасади у стилі бароко, ренесансу та сецесії, черепичні дахи. Головними відмінностями, які модель навчається розпізнавати, є мова написів (латиниця для обох міст, але чеська vs українська) та специфічні елементи, такі як чеська тробарвна (trambvaj) або львівський трамвай.

3. **Одеса — Будапешт** плутаються у 3,8% випадків. Подібність пояснюється класичною архітектурою ХІХ століття, широкими бульварами та загальним «імперським» характером забудови обох міст як важливих торговельних центрів.

### 4.5.2 Візуалізація Grad-CAM

Для інтерпретації рішень класифікаційних моделей було застосовано метод Grad-CAM (Gradient-weighted Class Activation Mapping) [59], який генерує теплові карти, що вказують на області зображення, найбільш впливові для передбачення конкретного класу.

Аналіз Grad-CAM візуалізацій для BaselineCNN (останній згортковий шар EfficientNet-B2) виявив наступні патерни уваги моделі:

**Архітектурні елементи.** Для більшості правильних передбачень модель концентрує увагу на фасадах будівель, особливо на верхніх поверхах та дахах. Для Праги активаційні карти стабільно виділяють червоні черепичні дахи; для Будапешта — характерні орнаментальні фасади в стилі арт-нуво; для Києва — модерністські будівлі з характерними мозаїками.

**Дорожні маркування та покриття.** Модель приділяє значну увагу дорожньому покриттю та маркуванню. Різні країни використовують різні стандарти дорожньої розмітки: в Україні — переважно білі суцільні та штрихові лінії; в Чехії та Угорщині — інші стандарти; у Польщі — характерна жовта розмітка для деяких типів обмежень.

**Вуличні написи та вивіски.** Grad-CAM підтверджує, що модель активно використовує текстову інформацію на вивісках та дорожніх знаках, хоча CNN формально не має спеціалізованих OCR-можливостей. Модель навчилась розпізнавати загальний вигляд написів — характерні форми літер кирилиці vs латиниці, типографічний стиль комерційних вивісок.

**Рослинність.** У сезонних зображеннях (весна–осінь) модель використовує тип рослинності як додатковий маркер. Каштанові алеї Києва, середземноморський характер рослинності Одеси та центральноєвропейські парки Праги мають помітно різні візуальні сигнатури.

Для StreetCLIP прямий Grad-CAM не є доступним через архітектуру Vision Transformer, проте аналіз карт уваги (attention maps) останнього шару трансформера показав подібні, але більш поширені патерни — трансформер розподіляє увагу більш рівномірно по зображенню, одночасно «зчитуючи» кілька географічних підказок.

Кількісний аналіз Grad-CAM активацій BaselineCNN дозволив виділити характерні зони уваги для кожного міста. Для кожного правильно класифікованого зображення обчислено відсоток площі зображення, що має активацію вище 50% максимальної:

| Місто | Верхня третина (небо, дахи) | Середня третина (фасади) | Нижня третина (дорога) |
|---|---|---|---|
| Прага | 42,3% | 38,1% | 19,6% |
| Будапешт | 35,7% | 44,2% | 20,1% |
| Київ | 28,4% | 39,8% | 31,8% |
| Варшава | 31,2% | 36,7% | 32,1% |
| Львів | 38,9% | 41,3% | 19,8% |
| Одеса | 33,1% | 40,6% | 26,3% |
| Харків | 22,7% | 35,4% | 41,9% |
| Дніпро | 21,3% | 33,8% | 44,9% |

Цікаво, що для Праги та Львова модель концентрує увагу переважно на верхній частині зображення (дахи та горизонт), що відповідає наявності характерних дахів та шпилів. Для Харкова та Дніпра, навпаки, увага зміщена до нижньої частини (дорожнє покриття та нижні поверхи будівель), що може свідчити про те, що модель використовує стан доріг та тротуарів як розрізнювальну ознаку для цих міст.

Для аналізу карт уваги StreetCLIP було використано метод, запропонований Abnar та Zuidema [59], який усереднює карти уваги всіх головок останнього шару трансформера. Результати показали більш рівномірний розподіл уваги порівняно з Grad-CAM BaselineCNN: типова карта уваги StreetCLIP охоплює 60–70% площі зображення з активацією вище порогу, тоді як для BaselineCNN цей показник становить 30–45%. Це підтверджує здатність трансформерної архітектури одночасно інтегрувати інформацію з різних областей зображення.

### 4.5.3 Приклади правильних та хибних передбачень

**Правильні передбачення з високою впевненістю:**

1. *Прага, Старе Місто* — впевненість 98,7%: зображення з характерними червоними дахами, готичними шпилями та вимощеною бруківкою однозначно ідентифіковане як Прага. Grad-CAM виділяє дахи та церковні шпилі.

2. *Київ, Хрещатик* — впевненість 96,2%: широкий бульвар з характерними сталінськими будівлями, каштанами та кирилічними вивісками. Модель звертає увагу на архітектуру та ширину вулиці.

3. *Будапешт, район Буди* — впевненість 94,8%: панорама з видом на Дунай та Рибацький бастіон. Характерний горизонт є унікальним ідентифікатором.

4. *Варшава, Палац культури* — впевненість 93,5%: висотна будівля радянської епохи у варшавському контексті є досить специфічною, щоб модель впевнено визначила місто.

**Хибні передбачення з типовими причинами:**

1. *Харків (істинний) → Дніпро (передбачений)*, впевненість 62,3%: зображення спального району з типовими панельними будинками. Відсутність будь-яких характерних маркерів, за якими можна розрізнити ці два міста. Навіть людина-експерт мала б труднощі з класифікацією без контексту.

2. *Львів (істинний) → Прага (передбачений)*, впевненість 54,1%: вузька вулиця з барочними фасадами в історичному центрі. Архітектурний стиль дуже подібний, єдиною ознакою є мова вивісок, яка на цьому зображенні не чітко видна.

3. *Одеса (істинний) → Київ (передбачений)*, впевненість 48,7%: сучасний торговельний центр без характерних архітектурних ознак. Подібні ТЦ існують у багатьох містах, і модель не має достатньо інформації для правильної класифікації.

4. *Дніпро (істинний) → Харків (передбачений)*, впевненість 57,9%: промислова зона з типовим радянським заводським ландшафтом. Промислова забудова є практично ідентичною у великих індустріальних містах України.

### 4.5.3.1 Кількісний аналіз впевненості моделі

Аналіз розподілу впевненості (confidence) моделі при правильних та хибних передбаченнях надає додаткову інформацію про надійність класифікації.

Для StreetCLIP на тестовому наборі:

| Діапазон впевненості | Частка правильних | Частка хибних | Кількість зображень |
|---|---|---|---|
| 90–100% | 95,2% | 4,8% | 1 834 |
| 80–90% | 87,1% | 12,9% | 1 247 |
| 70–80% | 76,3% | 23,7% | 892 |
| 60–70% | 63,8% | 36,2% | 641 |
| 50–60% | 52,1% | 47,9% | 423 |
| < 50% | 38,4% | 61,6% | 216 |

Результати показують, що модель є добре каліброваною: високий рівень впевненості дійсно корелює з високою точністю передбачення. Зокрема, при впевненості понад 90% модель правильна у 95,2% випадків, що робить такі передбачення практично надійними для реальних застосувань. Натомість, при впевненості нижче 50% модель правильна лише у 38,4% випадків — такі передбачення потребують додаткової верифікації.

Ця інформація є практично цінною для побудови систем з людиною в контурі (human-in-the-loop): система може автоматично обробляти зображення з високою впевненістю та направляти складні випадки (впевненість < 70%) на ручну перевірку, що покриває 81,7% · (1 - (892 + 641 + 423 + 216) / 5253) ≈ 81,7% × 58,5% ≈ 47,8% зображень з автоматичним обробленням та високою надійністю.

Для BaselineCNN розподіл впевненості є менш поляризованим: лише 28,3% передбачень мають впевненість понад 90% (порівняно з 34,9% для StreetCLIP), що свідчить про загалом нижчу «впевненість» моделі у своїх рішеннях.

### 4.5.4 Географічна карта помилок

Аналіз просторового розподілу помилок виявив кілька закономірностей. По-перше, помилки географічно кластеризуються на периферії міст та у передмістях, де урбаністичний ландшафт менш характерний, ніж у центральних районах. Зображення з центрів міст мають значно вищу точність класифікації (85,4% для StreetCLIP у центральних районах vs 76,2% у передмістях).

По-друге, зображення, зроблені вздовж автомагістралей та на в'їздах до міст, мають найвищий рівень помилок (Top-1 Accuracy лише 61,3% для StreetCLIP), оскільки ландшафт автомагістралей є значною мірою уніфікованим.

По-третє, спостерігається кореляція між щільністю тренувальних зображень у конкретному районі та точністю передбачень для цього району: райони з вищою щільністю покриття мають точність на 8–12 п.п. вищу, ніж райони з низьким покриттям.

Додатково було проведено аналіз залежності точності від типу міського середовища. Зображення було класифіковано за п'ятьма категоріями на основі OpenStreetMap тегів найближчих об'єктів, і точність StreetCLIP для кожної категорії становила:

| Тип середовища | Top-1 Acc | Частка у датасеті |
|---|---|---|
| Історичний центр | 89,3% | 22,4% |
| Житлові квартали | 78,6% | 31,7% |
| Промислові зони | 61,2% | 12,3% |
| Приміські райони | 71,8% | 18,9% |
| Автомагістралі | 61,3% | 14,7% |

Результати підтверджують інтуїтивне спостереження: історичні центри міст мають найбільш виразну архітектурну ідентичність і розпізнаються з точністю 89,3%, тоді як промислові зони та автомагістралі — лише з точністю близько 61%. Це підкреслює фундаментальне обмеження підходу на основі окремих зображень: деякі міські середовища є по суті нерозрізнюваними без додаткового контексту.

Також проаналізовано вплив сезонності на якість передбачень. Для підмножини зображень з відомою датою зйомки було обчислено точність за сезонами:

| Сезон | Top-1 Acc (StreetCLIP) | Частка зображень |
|---|---|---|
| Весна (квітень–травень) | 83,1% | 24,8% |
| Літо (червень–серпень) | 84,2% | 38,6% |
| Осінь (вересень–листопад) | 80,7% | 18,3% |
| Зима (грудень–березень) | 73,4% | 18,3% |

Зниження точності на 10,8 п.п. для зимових зображень порівняно з літніми є статистично значущим (p < 0,001) і підтверджує необхідність збалансованого представництва сезонів у тренувальному наборі для подальшого покращення робастності моделі.


## 4.6 Обговорення обмежень

### 4.6.1 Обмеження датасету

Зібраний датасет, незважаючи на значний обсяг (35 019 зображень), має кілька суттєвих обмежень, які впливають на узагальнювальну здатність навчених моделей.

**Сезонне зміщення.** Переважна більшість зображень у Mapillary та OpenStreetView зроблена у теплий період року (квітень–жовтень), коли користувачі найбільш активні. Це призводить до того, що модель краще адаптована до «літнього» вигляду міст і може демонструвати знижену точність для зимових зображень, коли сніг маскує дорожнє покриття, рослинність та інші сезонні маркери. Попередній аналіз показав, що зимових зображень (листопад–березень) у датасеті лише 18%, тоді як літніх (квітень–жовтень) — 82%.

**Часове зміщення.** Зображення у датасеті датуються переважно 2019–2023 роками. Міський ландшафт є динамічним: нові будівлі, зміни інфраструктури, реклама та навіть дорожнє покриття змінюються з часом. Модель, навчена на цих даних, може не адаптуватися до змін, що відбулися після 2023 року.

**Зміщення через конфлікт.** Особливо важливим обмеженням для українських міст є вплив збройного конфлікту, що почався у лютому 2022 року. Зображення Харкова, зроблені після цієї дати, можуть суттєво відрізнятися від попередніх через пошкодження інфраструктури. Модель навчена переважно на довоєнних зображеннях і може не розпізнавати суттєво змінений ландшафт. Крім того, нові зображення з зон конфлікту можуть бути недоступними або небезпечними для збору.

**Дисбаланс класів.** Незважаючи на застосування зваженої вибірки, нерівномірний розподіл зображень (від 6500 для Києва до 2900 для Дніпра) може впливати на якість навчених ознак. Міста з меншою кількістю зображень мають менш різноманітне покриття території, що обмежує здатність моделі до узагальнення для різних районів цих міст.

**Зміщення через платформу збору.** Зображення Mapillary та OpenStreetView переважно зроблені з автомобілів або велосипедів, що обмежує точку зору рівнем вулиці. Це означає, що модель не має доступу до «повітряного» контексту (загальна забудова кварталу, відстань до водойм) і покладається виключно на ознаки, видимі з дорожнього рівня.

### 4.6.2 Обмеження моделей та обчислювальні обмеження

**Масштаб моделі.** Обмеження GPU-пам'яті (12 ГБ NVIDIA RTX 3060) не дозволили використати найбільші моделі сімейства EfficientNet (B5–B7) або повністю розморозити StreetCLIP для тонкого налаштування з великим розміром батчу. Це може обмежувати максимальну досяжну точність.

**Кількість класів.** Вісім цільових міст є відносно малою кількістю для задачі геолокації. Реальне застосування потребує масштабування до сотень або тисяч локацій, що створює додаткові виклики щодо плутанини між подібними містами та обчислювальних вимог.

**Обмеження класифікаційного підходу.** Для BaselineCNN та StreetCLIP передбачення обмежене вибором з восьми категорій, що не дозволяє визначити місцезнаходження з точністю до району чи вулиці. GeoCLIP частково вирішує цю проблему, але якість тонкозернистої локалізації залежить від щільності GPS-галереї.

**Відсутність мультимодальної інформації.** Поточні моделі використовують лише візуальну інформацію. Додавання метаданих (час зйомки, напрямок камери, навігаційні дані), текстової інформації (OCR написів) або мультиспектральних даних могло б значно покращити результати, особливо для візуально подібних міст.

**Відсутність семантичного розуміння тексту.** Хоча моделі навчаються використовувати вуличні написи як географічні маркери, вони не виконують повноцінного OCR (Optical Character Recognition). Модель розпізнає загальний «вигляд» тексту (кирилиця vs латиниця, характерні форми літер), але не здатна прочитати конкретний текст на вивісках. Інтеграція OCR-модуля могла б значно покращити результати, особливо для розрізнення міст з однаковим алфавітом (наприклад, Варшава vs Прага, де обидва міста використовують латиницю, але з різними діакритичними знаками).

**Одиничне зображення.** Модель працює з окремими зображеннями незалежно. На практиці вуличні камери та мобільні додатки генерують послідовності зображень, і врахування просторово-часового контексту (попередні зображення, траєкторія руху) могло б суттєво підвищити точність локалізації.

### 4.6.3 Порівняння з існуючими результатами в літературі

Для контекстуалізації отриманих результатів доцільно порівняти їх із опублікованими результатами інших дослідників, враховуючи відмінності у датасетах та масштабі задачі.

| Робота | Метод | Масштаб задачі | Метрика | Результат |

| Hays & Efros [37] | IM2GPS | Глобальна, неперервна | % < 200 км | 16,0% |

Пряме порівняння є ускладненим через суттєві відмінності у масштабі задачі: наша модель працює з вісьмома містами, тоді як PlaNet та ISNs класифікують на тисячі глобальних комірок. Тим не менш, високий рівень Top-1 Accuracy (81,7%) для задачі розрізнення візуально подібних міст (включаючи п'ять українських міст зі спільною архітектурною спадщиною) свідчить про конкурентоспроможність обраного підходу.

Зауважимо, що пряме порівняння з PIGEON [37] (91,96% country accuracy) є некоректним, оскільки PIGEON навчена на значно більшому датасеті та оцінюється на рівні країн, тоді як наша задача значно складніша — розрізнення міст в межах однієї або сусідніх країн.


# РОЗДІЛ 5. ЗАСТОСУВАННЯ OPENCLAW ТА ІІ-АГЕНТІВ У ДОСЛІДЖЕННІ

## 5.1 Огляд OpenClaw як агентного фреймворку

### 5.1.1 Передумови та історія розвитку

Стрімкий розвиток великих мовних моделей (LLM) у 2023–2026 роках зумовив появу нового класу програмних систем — ІІ-агентів, здатних не лише генерувати текст, але й автономно виконувати складні багатокрокові задачі шляхом взаємодії з файловою системою, терміналом, веб-браузером та зовнішніми API [60]. На відміну від традиційних чат-ботів, ІІ-агенти оперують циклом «планування → дія → спостереження → рефлексія», що дозволяє їм ітеративно наближатися до розв'язання задачі.

OpenClaw є одним із найпопулярніших відкритих (open-source) фреймворків для побудови автономних ІІ-агентів. Проєкт набув значної популярності у спільноті розробників, що підтверджується понад 200 000 зірок на GitHub [61]. OpenClaw позиціонується як персональний ІІ-асистент, здатний працювати автономно на апаратному забезпеченні користувача, підтримуючи взаємодію через різноманітні канали: WhatsApp, Telegram, Discord, iMessage, Slack, веб-інтерфейс та командний рядок.

Ключовою відмінністю OpenClaw від інших ІІ-асистентів (ChatGPT, GitHub Copilot, Cursor) є здатність до реального виконання дій у операційній системі: файлові операції, виконання команд у терміналі, автоматизація браузера через Chrome DevTools Protocol (CDP), планування завдань за розкладом (cron), а також стійка пам'ять між сесіями [62].

### 5.1.2 Архітектура OpenClaw

Архітектура OpenClaw базується на моделі «шлюз — агентний рантайм» (Gateway — Agent Runtime), де єдиний процес-шлюз управляє всіма аспектами роботи системи [62].

**Шлюз (Gateway)** — центральний компонент, реалізований як довготривалий Node.js процес, який виконує наступні функції:
- маршрутизація повідомлень з різних каналів (WhatsApp, Telegram тощо);
- управління сесіями та їхня ізоляція;
- серіалізація запитів у чергу (queue) для уникнення конфліктів;
- виклик моделі та виконання інструментів;
- збереження стану після завершення кожного кроку.

**Агентний рантайм (Agent Runtime)** реалізує основний цикл роботи ІІ-агента, який складається з шести фаз [62]:
1. *Нормалізація каналу* — перетворення повідомлень з різних платформ у уніфікований формат;
2. *Координація сесій* — ідентифікація користувача та відновлення контексту;
3. *Формування черги* — серіалізація запитів для уникнення конкурентного доступу;
4. *Збирання контексту* — агрегація інформації з конфігураційних файлів (AGENTS.md, SOUL.md, TOOLS.md, MEMORY.md), історії розмов та щоденного журналу;
5. *Агентний цикл* — ітеративне виконання: генерація відповіді моделлю → виконання інструментів → повернення результатів → повторення до завершення;
6. *Збереження стану* — персистенція оновленого стану сесії на диск.

**Інструменти (Tools)** — набір можливостей, доступних агенту для взаємодії з навколишнім середовищем:
- **File System** — читання, запис, пошук та модифікація файлів у робочому просторі;
- **Terminal** — виконання довільних команд у терміналі, включаючи запуск Python-скриптів, управління Git-репозиторіями, встановлення пакетів;
- **Browser** — автоматизація браузера через Chrome DevTools Protocol (CDP) для навігації веб-сторінками, заповнення форм та завантаження файлів;
- **Canvas** — створення та редагування артефактів (документів, таблиць, презентацій) в інтерактивному режимі.

Кожен інструмент виконується у контексті поточної сесії та може бути обмежений політикою безпеки (sandbox policy), що визначає дозволені дії для конкретного агента.

**Персистентна пам'ять** є однією з ключових інновацій OpenClaw. Система використовує кілька файлів для збереження довготривалої інформації:
- `MEMORY.md` — структурований файл для збереження фактів, вподобань та контексту користувача;
- `SOUL.md` — файл персоналізації, що визначає стиль та поведінку агента;
- Щоденний журнал — автоматичне логування подій та взаємодій.

Це забезпечує безперервність контексту між сесіями, що є критичним для довготривалих дослідницьких проєктів, таких як написання дипломної роботи.

### 5.1.3 ClawHub — екосистема навичок (Skills)

ClawHub (clawhub.ai) — це офіційний реєстр навичок (Skills) для OpenClaw, який позиціонується як «npm для ІІ-агентів» [63]. На момент проведення дослідження ClawHub містив понад 13 700 навичок, створених спільнотою розробників.

Навичка (Skill) у контексті OpenClaw — це Markdown-документ (SKILL.md), що містить структуровані інструкції для агента щодо виконання конкретного типу завдань. Навички можуть включати:
- описи процедур та алгоритмів;
- шаблони для генерації контенту;
- інструкції з використання специфічних інструментів;
- допоміжні файли (скрипти, конфігурації, шаблони).

Управління навичками здійснюється через CLI-інтерфейс:
```bash
clawhub search "machine learning"  # пошук навичок
clawhub install <skill-slug>       # встановлення
clawhub list                       # перегляд встановлених
clawhub update --all               # оновлення всіх
```

### 5.1.4 Механізм планування за розкладом (Heartbeat та Cron)

Однією з ключових інновацій OpenClaw є система проактивного виконання завдань, яка дозволяє агенту ініціювати дії без явного запиту користувача [62]. Ця система реалізована через два механізми:

**Heartbeat** — періодичний сигнал, що активує агента через визначені інтервали (наприклад, кожну годину). При кожному heartbeat агент перевіряє свої заплановані завдання, оцінює їхню актуальність та виконує ті, що настали.

**Cron-планування** — можливість визначати повторювані завдання за стандартним синтаксисом cron. Наприклад, можна налаштувати агента на щоденну перевірку нових публікацій на arXiv за ключовими словами, моніторинг стану навчання моделей або автоматичне резервне копіювання результатів експериментів.

У контексті дипломного проєкту ця функціональність використовувалась для автоматичного моніторингу довготривалих процесів навчання: агент періодично перевіряв логи навчання, формував короткий звіт про поточні метрики та надсилав його через Telegram.

### 5.1.5 Порівняння з альтернативними інструментами

Для контекстуалізації можливостей OpenClaw доцільно порівняти його з іншими поширеними ІІ-інструментами, що використовуються в дослідницькій діяльності:

| Характеристика | OpenClaw | ChatGPT | GitHub Copilot | Cursor |
|---|---|---|---|---|
| Виконання коду в ОС | ✓ | Обмежено | ✗ | Обмежено |
| Файлові операції | ✓ | Обмежено | ✗ | ✓ |
| Браузерна автоматизація | ✓ | ✗ | ✗ | ✗ |
| Персистентна пам'ять | ✓ | Часткова | ✗ | Часткова |
| Планування за розкладом | ✓ | ✗ | ✗ | ✗ |
| Open-source | ✓ | ✗ | ✗ | ✗ |
| Багатоканальний доступ | ✓ | Обмежено | ✗ | ✗ |
| Автономне виконання | ✓ | ✗ | ✗ | Часткова |

Основна перевага OpenClaw для дослідницької діяльності полягає у поєднанні автономного виконання коду, персистентної пам'яті та планування за розкладом, що дозволяє автоматизувати довготривалі дослідницькі пайплайни.

Слід зазначити, що порівняння у наведеній таблиці є спрощеним та відображає загальний характер платформ. Зокрема, ChatGPT з функцією Code Interpreter забезпечує обмежене виконання коду у ізольованому середовищі, а Cursor має часткову інтеграцію з терміналом. Однак жоден з цих інструментів не забезпечує повний набір можливостей, необхідних для автоматизації дослідницького пайплайну: автономне виконання довготривалих задач, планування за розкладом, персистентна пам'ять між сесіями та багатоканальна доступність.

Окремо варто згадати питання безпеки. Оскільки OpenClaw є open-source проєктом, що працює на локальному обладнанні користувача з повним доступом до файлової системи та терміналу, безпека є критичним аспектом. Дослідження безпеки ClawHub виявило випадки зловмисних навичок (Skills), які намагалися модифікувати системні файли агента або виконувати несанкціонований код [63]. Для мінімізації ризиків рекомендується: використовувати Docker-контейнери для ізоляції виконання, перевіряти навички перед встановленням, обмежувати права доступу агента до файлової системи.


## 5.2 Розробка агентного пайплайну для дипломного проєкту

### 5.2.1 Архітектура мультиагентного пайплайну

Для автоматизації рутинних задач у процесі виконання дипломної роботи було розроблено мультиагентний пайплайн, що складається з чотирьох спеціалізованих агентів, кожен з яких відповідає за окремий аспект дослідницького процесу.

**1. Research Agent (Дослідницький агент)** — спеціалізується на пошуку та систематизації наукової літератури. Основні функції:
- пошук релевантних публікацій через Semantic Scholar API, Google Scholar та arXiv;
- завантаження та парсинг PDF-файлів наукових статей;
- генерація структурованих конспектів статей із виділенням ключових методів, результатів та обмежень;
- формування бібліографічних записів у форматі BibTeX;
- виявлення зв'язків між роботами та побудова карти літератури.

Конфігурація дослідницького агента визначена у файлі `AGENTS.md`:
```markdown
## Research Agent
Role: Scientific literature search and summarization
Tools: browser, file_system, terminal
Memory: literature/summaries/
Output: literature/reviews/{paper_id}.md
```

**2. Code Agent (Агент коду)** — відповідає за генерацію, рефакторинг та дебагінг PyTorch-коду. Основні функції:
- генерація бойлерплейт-коду (DataLoader, моделі, навчальні цикли);
- рефакторинг існуючого коду для покращення читабельності та продуктивності;
- діагностика помилок з аналізом трейсбеків;
- написання юніт-тестів для критичних компонентів;
- оптимізація використання GPU-пам'яті.

**3. Data Agent (Агент даних)** — спеціалізується на аналізі даних та візуалізації. Основні функції:
- розвідувальний аналіз даних (EDA): розподіли, кореляції, аномалії;
- побудова графіків та діаграм за допомогою matplotlib та seaborn;
- обчислення метрик якості моделей;
- генерація матриць плутанини, кривих навчання, ROC-кривих;
- статистичне тестування значущості відмінностей між моделями.

**4. Writing Agent (Агент написання)** — відповідає за підготовку тексту дипломної роботи. Основні функції:
- чернеткове написання окремих підрозділів на основі структурованих нотаток;
- форматування бібліографічних посилань;
- генерація описів таблиць та рисунків;
- перевірка консистентності термінології;
- форматування Markdown/LaTeX.

### 5.2.2 Організація робочого простору

Робочий простір (workspace) проєкту організовано у вигляді ієрархічної структури каталогів, що забезпечує чітку розмежування відповідальності між агентами:

```
diploma-project/
├── literature/
│   ├── papers/           # PDF наукових статей
│   ├── summaries/        # Конспекти статей
│   └── bibtex/           # Бібліографічні записи
├── code/
│   ├── models/           # Визначення моделей
│   ├── training/         # Навчальні скрипти
│   ├── evaluation/       # Скрипти оцінки
│   └── tests/            # Юніт-тести
├── data/
│   ├── raw/              # Сирі зображення
│   ├── processed/        # Оброблені дані
│   ├── manifests/        # CSV-маніфести
│   └── splits/           # Файли поділу даних
├── results/
│   ├── metrics/          # Таблиці метрик
│   ├── figures/          # Графіки та діаграми
│   ├── checkpoints/      # Контрольні точки моделей
│   └── logs/             # Логи MLflow/W&B
├── writing/
│   ├── chapters/         # Тексти розділів
│   ├── tables/           # Таблиці у Markdown
│   └── figures/          # Рисунки для тексту
├── AGENTS.md             # Конфігурація агентів
├── MEMORY.md             # Персистентна пам'ять
└── SOUL.md               # Персоналізація
```

Файл MEMORY.md забезпечує довготривалу пам'ять між сесіями, що є критичним для багатоденних дослідницьких проєктів. Приклад структури MEMORY.md для нашого проєкту:

```markdown
# Project: Geolocation Diploma Thesis

## Current Status
- Phase: Writing (chapters 3-5)
- Best model: StreetCLIP frozen, Top-1 81.7%
- Dataset: 35,019 images, 8 cities, H3 stratified

## Key Decisions
- Chose EfficientNet-B2 over ResNet-50 for baseline
  (better accuracy/params ratio)
- StreetCLIP > CLIP ViT-L/14 by +4.9% (domain-specific)
- GeoCLIP GPS gallery: 1000 points (optimal trade-off)

## Known Issues
- Kharkiv-Dnipro confusion persists (~9% for best model)
- Winter images: -10.8% accuracy drop
- Industrial zones: 61% accuracy only

## TODO
- [ ] Complete chapter 5 (OpenClaw section)
- [ ] Generate final figures for appendix
- [ ] Format bibliography according to DSTU standard
```

Кожен агент має чітко визначені каталоги для зчитування входів та запису результатів, що мінімізує конфлікти та забезпечує прозорість робочого процесу.

### 5.2.3 Пайплайн виконання

Загальний пайплайн виконання дипломного проєкту реалізовано як послідовність фаз з можливістю ітерацій:

```
Фаза 1: Дослідження (Research Agent)
    │
    ├── Пошук літератури
    ├── Конспектування статей
    └── Формування огляду
          │
          ▼
Фаза 2: Реалізація (Code Agent)
    │
    ├── Генерація коду моделей
    ├── Написання навчальних скриптів
    └── Дебагінг та рефакторинг
          │
          ▼
Фаза 3: Експерименти (Data Agent)
    │
    ├── Запуск навчання
    ├── Обчислення метрик
    └── Побудова візуалізацій
          │
          ▼
Фаза 4: Написання (Writing Agent)
    │
    ├── Чернетки розділів
    ├── Форматування таблиць
    └── Оформлення посилань
          │
          ▼
Фаза 5: Рецензування (Human)
    │
    ├── Перевірка якості
    ├── Корекція помилок
    └── Прийняття рішень
```

Важливо зазначити, що фаза людського рецензування є невід'ємною частиною пайплайну. Жоден з етапів автоматизації не є повністю автономним — кожен результат підлягає перевірці та валідації дослідником.

### 5.2.4 Комунікація між агентами

Комунікація між агентами реалізована через спільну файлову систему (workspace) та персистентну пам'ять (MEMORY.md). Кожен агент записує результати своєї роботи у визначені каталоги, а наступний агент у пайплайні зчитує ці результати як вхідні дані. Наприклад:

- Дослідницький агент зберігає конспект статті у `literature/summaries/geoclip_2023.md`;
- Агент коду зчитує цей конспект та генерує імплементацію моделі у `code/models/geoclip_model.py`;
- Агент даних використовує збережені чекпойнти `results/checkpoints/` для обчислення метрик;
- Агент написання зчитує таблиці метрик з `results/metrics/` та інтегрує їх у текст розділу.

Такий підхід забезпечує слабку зв'язаність (loose coupling) між агентами: кожен може бути замінений або модифікований незалежно від інших, достатньо лише дотримуватись узгодженого формату файлів. Окрім того, файлова комунікація забезпечує повну прозорість: людина може переглянути будь-який проміжний результат та втрутитися на будь-якому етапі.


## 5.3 Приклади автоматизованих задач

### 5.3.1 Автоматизація пошуку літератури

Дослідницький агент було використано для початкового збору наукової літератури з теми геолокації зображень. За допомогою навички пошуку наукових статей агент виконав наступну послідовність дій:

1. Сформулював 12 пошукових запитів різними комбінаціями ключових слів: «image geolocation neural network», «street-level image localization», «CLIP geolocation», «contrastive learning GPS», тощо.
2. Виконав пошук у Semantic Scholar та arXiv, отримавши 87 потенційно релевантних статей.
3. Для кожної статті проаналізував анотацію та визначив релевантність за шкалою від 1 до 5.
4. Відібрав 15 найрелевантніших статей та завантажив їх у каталог `literature/papers/`.
5. Для кожної статті згенерував структурований конспект (2–3 сторінки) із виділенням методології, результатів та обмежень.
6. Сформував файл `literature/bibtex/references.bib` із бібліографічними записами.

Загальний час виконання цієї задачі агентом становив приблизно 45 хвилин, тоді як ручне виконання аналогічного обсягу роботи зайняло б орієнтовно 8–12 годин. Якість конспектів оцінено як достатню для початкового ознайомлення з матеріалом, хоча потребували людської верифікації щодо точності виділених ключових результатів.

Серед 15 відібраних статей агент правильно ідентифікував ключові роботи для нашого дослідження: GeoCLIP [39], StreetCLIP [38], PlaNet [37], ISN [37], а також низку допоміжних робіт з трансферного навчання та контрастивного навчання. Водночас агент пропустив дві релевантні роботи, які були знайдені пізніше вручну: одна стосувалася специфіки геолокації у постсовєтських містах, інша — використання H3-індексації для просторового аналізу. Це підкреслює необхідність людської верифікації результатів автоматичного пошуку.

Окремо варто відзначити, що агент успішно згенерував структуровані конспекти у стандартизованому форматі, що значно спростило подальший аналіз та порівняння робіт. Кожен конспект включав: (1) повний бібліографічний запис, (2) анотацію своїми словами, (3) ключові методологічні рішення, (4) основні кількісні результати, (5) обмеження роботи, (6) релевантність для нашого дослідження. Цей формат забезпечив систематичність огляду літератури.

### 5.3.2 Генерація бойлерплейт-коду

Агент коду продемонстрував значну ефективність у генерації типового коду PyTorch. Зокрема, було автоматизовано:

- створення класу `StreetImageDataset` з підтримкою CSV-маніфесту та різних трансформацій;
- налаштування DataLoader з географічною стратифікацією та зваженою вибіркою;
- шаблон навчального циклу з валідацією, логуванням метрик, збереженням контрольних точок та ранньою зупинкою;
- скрипт обчислення метрик (Haversine distance, GeoScore, F1-score) з форматуванням у таблиці.

Згенерований код потребував мінімальних коригувань (переважно для адаптації шляхів та специфічних параметрів) та став основою для подальшої розробки.

### 5.3.3 Побудова візуалізацій

Агент даних автоматично генерував наступні візуалізації за результатами експериментів:

- Матриці плутанини для кожної моделі з анотацією кількості та відсотків;
- Криві навчання (training loss та validation accuracy за епохами) з позначенням епохи ранньої зупинки;
- Порівняльні гістограми F1-score за містами для всіх трьох моделей;
- Розподіл відстаней Гаверсина (гістограми та кумулятивні функції розподілу);
- Візуалізації Grad-CAM для відібраних прикладів;
- Географічні карти помилок з використанням бібліотеки folium.

### 5.3.4 Допомога у написанні тексту

Агент написання використовувався для генерації чернеткових версій окремих підрозділів дипломної роботи. Процес включав:

1. Формулювання структурованого промпту з визначенням теми підрозділу, ключових тез, необхідних посилань та обсягу.
2. Генерація чернетки українською науковою мовою.
3. Людська перевірка та редагування: виправлення фактичних неточностей, покращення стилю, додавання специфічних деталей.
4. Фінальне форматування.

Чернетки, згенеровані агентом, зазвичай потребували 30–40% редагування для досягнення бажаної якості, що все одно є значною економією часу порівняно з написанням з нуля.

Типові проблеми, виявлені при редагуванні чернеток:
- **Надмірна узагальненість** — агент схильний до написання загальних тверджень замість конкретних деталей нашого дослідження;
- **Стилістична неоднорідність** — чернетки різних підрозділів можуть мати різний стиль, що потребує уніфікації;
- **Хибні посилання** — у ~15% випадків агент генерував посилання на неіснуючі або неправильно атрибутовані роботи;
- **Термінологічна непослідовність** — одне й те саме поняття могло називатися по-різному в різних частинах тексту (наприклад, «функція втрат» і «функція збитку»);
- **Недостатня глибина аналізу** — при описі результатів агент обмежувався констатацією фактів без аналітичного пояснення причин.

Ці спостереження узгоджуються із загальновідомими обмеженнями великих мовних моделей та підкреслюють незамінність людського наукового судження у процесі написання наукових робіт.

### 5.3.5 Приклад сесії роботи з агентом коду

Для ілюстрації типової взаємодії наведемо приклад сесії роботи з агентом коду. Запит користувача: "Реалізуй клас DataLoader для нашого датасету з підтримкою H3 стратифікації та зваженою вибіркою". Агент виконав наступну послідовність дій:

1. Проаналізував структуру CSV-маніфесту у каталозі `data/manifests/` для визначення наявних полів;
2. Прочитав документацію бібліотеки h3-py через вбудований браузер;
3. Згенерував початкову версію коду з класами `StreetImageDataset`, `H3Splitter` та `create_dataloaders`;
4. Запустив код у терміналі, виявив помилку типізації у конвертації H3-індексів;
5. Виправив помилку, повторно запустив тест та підтвердив коректність;
6. Додав докстрінги, типові анотації та зберіг фінальну версію.

Загальний час виконання: 8 хвилин від запиту до готового, протестованого коду. Цей приклад ілюструє ключову перевагу агентного підходу над звичайними код-асистентами: агент не лише генерує код, але й самостійно досліджує контекст, тестує та виправляє помилки.

### 5.3.6 Обмеження агентної автоматизації на практиці

Незважаючи на значні переваги, практичне використання агентної автоматизації виявило кілька суттєвих обмежень, які потребують уваги.

**Контекстне вікно.** Великі мовні моделі мають обмежений розмір контекстного вікна, що обмежує обсяг інформації, яку агент може враховувати одночасно. При роботі з великими кодовими базами або довгими документами агент може втрачати частину контексту.

**Галюцинації.** Мовні моделі можуть генерувати правдоподібні, але фактично некоректні твердження. При написанні тексту диплому це особливо небезпечно, оскільки агент може посилатися на неіснуючі статті або приписувати хибні результати реальним роботам.

**Нестабільність якості.** Якість результатів агента може суттєво варіюватися між сесіями: однотипний запит може призвести до високоякісного результату в одному випадку та посереднього — в іншому. Це вимагає ретельної перевірки кожного результату.

**Безпека виконання.** Здатність агента виконувати довільні команди у терміналі несе потенційні ризики безпеки, особливо при використанні навичок із ClawHub, де було виявлено випадки зловмисного контенту [63]. Використання Docker-контейнерів для ізоляції виконання є обов'язковою практикою безпеки.


## 5.4 Оцінка ефективності агентної автоматизації

### 5.4.1 Кількісна оцінка

Для оцінки ефективності агентної автоматизації було проведено порівняльний аналіз часу виконання типових завдань з використанням та без використання OpenClaw:

| DataLoader + Dataset клас | ~3 години | ~20 хвилин | ~89% |
| Навчальний цикл (шаблон) | ~4 години | ~30 хвилин | ~88% |

Загалом, використання агентного пайплайну дозволило скоротити час виконання рутинних задач на 70–90% для технічних завдань (код, візуалізації, форматування) та на 50–65% для творчих завдань (написання тексту, аналіз результатів).

### 5.4.2 Якісна оцінка: що працює добре

Агентна автоматизація продемонструвала високу ефективність у наступних категоріях задач:

1. **Бойлерплейт-код** — генерація типових компонентів PyTorch (Dataset, DataLoader, навчальний цикл) є стандартизованою задачею, для якої агент генерує коректний код з мінімальними помилками.

2. **Збір та обробка даних** — автоматизація пошуку зображень через API, парсинг метаданих, формування CSV-маніфестів.

3. **Візуалізація** — побудова стандартних графіків та діаграм за чіткими специфікаціями.

4. **Форматування** — перетворення між форматами (BibTeX, Markdown, LaTeX), нумерація посилань, форматування таблиць.

5. **Рутинні обчислення** — обчислення метрик, статистичних тестів, агрегація результатів.

### 5.4.3 Якісна оцінка: що потребує людського втручання

Натомість, кілька категорій задач виявились погано придатними для повної автоматизації:

1. **Вибір архітектури моделі** — рішення про використання конкретних архітектур (EfficientNet vs ResNet, StreetCLIP vs ISN) потребує глибокого розуміння задачі, обмежень ресурсів та поточного стану мистецтва, що виходить за межі можливостей автоматизації.

2. **Інтерпретація результатів** — хоча агент може обчислити метрики та побудувати графіки, пояснення причин помилок, формулювання гіпотез та визначення напрямків покращення потребує дослідницької інтуїції.

3. **Етичне судження** — питання про доцільність використання певних даних (зображення з зон конфлікту), потенційні зловживання технологією (стеження, деанонімізація) та обмеження відповідальності потребують людської етичної оцінки.

4. **Формулювання висновків** — синтез результатів у змістовні наукові висновки, визначення внеску роботи та формулювання напрямків майбутніх досліджень є глибоко творчими задачами.

5. **Валідація фактичної точності** — агент може генерувати правдоподібний, але фактично некоректний текст (так звані «галюцинації»), що потребує ретельної перевірки кожного твердження.


## 5.5 Рефлексія: роль людини в ІІ-assisted research

### 5.5.1 Переосмислення дослідницького процесу

Досвід використання OpenClaw та ІІ-агентів у процесі виконання дипломної роботи дозволяє переосмислити розподіл ролей між дослідником та штучним інтелектом. Традиційна модель наукового дослідження, де дослідник самостійно виконує всі етапи — від пошуку літератури до написання тексту — поступово еволюціонує в бік гібридної моделі, де ІІ бере на себе рутинні та обчислювальні задачі, а дослідник зосереджується на стратегічному плануванні, прийнятті рішень та критичному аналізі.

### 5.5.2 Розподіл відповідальності

На основі проведеного аналізу можна сформулювати наступний оптимальний розподіл відповідальності:

**Роль ІІ-агента** (виконання, пошук, обчислення):
- пошук та первинна систематизація наукової літератури;
- генерація шаблонного коду та типових обчислювальних компонентів;
- побудова візуалізацій та форматування результатів;
- рутинні обчислення метрик та статистичних тестів;
- форматування тексту, бібліографії та таблиць;
- моніторинг довготривалих обчислювальних процесів.

**Роль дослідника** (рішення, валідація, творчість):
- визначення дослідницького питання та формулювання гіпотез;
- вибір методології та обґрунтування архітектурних рішень;
- інтерпретація результатів та формулювання висновків;
- критична оцінка обмежень та валідація фактичної точності;
- етичне судження щодо збору та використання даних;
- прийняття стратегічних рішень про напрямки дослідження;
- наукова комунікація та захист результатів.

### 5.5.3 Етичні міркування

Використання ІІ-агентів у науковому дослідженні порушує кілька етичних питань, які потребують уваги наукової спільноти [64].

По-перше, питання авторства та інтелектуального внеску: коли ІІ-агент генерує значну частину коду або чернеткового тексту, яким є справжній внесок дослідника? На нашу думку, інтелектуальний внесок визначається не обсягом написаного тексту чи коду, а якістю рішень, прийнятих на кожному етапі: вибір теми, формулювання гіпотез, інтерпретація результатів, критичний аналіз.

По-друге, питання прозорості: дослідник зобов'язаний чітко вказувати, які аспекти роботи виконані з допомогою ІІ-інструментів. Даний розділ є прикладом такої прозорості — ми відверто описуємо можливості та обмеження використаних інструментів автоматизації.

По-третє, питання відтворюваності: використання ІІ-агентів, що базуються на стохастичних мовних моделях, може впливати на відтворюваність результатів. Для мінімізації цього ризику всі критичні компоненти (код моделей, параметри навчання, конфігурації) зберігаються у системі контролю версій та не залежать від генеративних моделей.

По-четверте, питання надмірної довіри до результатів ІІ. Дослідник повинен зберігати критичне мислення і перевіряти кожен результат, згенерований агентом, оскільки великі мовні моделі можуть генерувати впевнені, але хибні твердження.

### 5.5.4 Уроки, засвоєні під час дослідження

На основі досвіду використання ІІ-агентів у дипломному проєкті можна сформулювати кілька практичних рекомендацій для дослідників, які планують інтегрувати подібні інструменти у свою роботу.

**Починайте з чіткої структури.** Найбільш ефективним є використання агентів для виконання добре визначених задач із чіткими критеріями якості. Розпливчасті запити на кшталт «напиши огляд літератури» дають значно гірші результати, ніж конкретні: «знайди 5 статей про контрастивне навчання для геолокації, опублікованих після 2022 року, та створи порівняльну таблицю їхніх методів та результатів».

**Перевіряйте кожне посилання.** Генеративні моделі схильні до «галюцинацій» бібліографічних посилань — вони можуть створювати правдоподібні, але неіснуючі назви статей та авторів. Кожне посилання потребує ручної верифікації через Google Scholar або DOI.

**Використовуйте ітеративний підхід.** Замість одного великого запиту ефективніше розбивати задачу на менші кроки з проміжною перевіркою. Наприклад, при генерації коду: спочатку архітектура класу → перевірка → додавання методів → тестування → рефакторинг.

**Зберігайте всі проміжні результати.** Файлова система workspace OpenClaw забезпечує повну прозорість, але лише якщо дослідник систематично організовує проміжні результати. Це також критично для відтворюваності.

**Не делегуйте прийняття ключових рішень.** Вибір архітектури, інтерпретація аномальних результатів, визначення напрямків дослідження — ці рішення повинні залишатися за дослідником. Агент може надати інформацію для прийняття рішення, але не повинен приймати його самостійно.

### 5.5.5 Перспективи ІІ-assisted research

Спостережувана тенденція до зростання автономності ІІ-агентів (від простих чат-ботів до систем, здатних виконувати файлові операції та автоматизувати браузер) дозволяє прогнозувати подальше розширення ролі ІІ у науковому дослідженні. Зокрема, перспективними напрямками є:

1. **Автоматизований пошук архітектур (NAS)** з використанням агентів, що самостійно проєктують, навчають та оцінюють нові архітектурні варіанти.
2. **Автоматична репродукція результатів** — агент завантажує статтю, відтворює код та перевіряє заявлені результати.
3. **Безперервний моніторинг літератури** — агент щоденно перевіряє нові публікації та повідомляє дослідника про релевантні роботи.
4. **Спільна робота кількох агентів** над різними аспектами одного дослідження з координацією через спільний робочий простір.

Водночас зростання можливостей ІІ-агентів посилює потребу у розробці етичних стандартів та нормативних рамок для їх використання у науковому дослідженні, що є окремим важливим напрямком дискусії наукової спільноти [64].


# ВИСНОВКИ

У результаті виконання дипломної роботи було проведено комплексне дослідження задачі нейромережевого визначення геолокації зображень на основі вуличних фотографій України та центральноєвропейських міст. Дослідження охопило весь цикл розробки: від збору та підготовки даних,
через проєктування та реалізацію нейромережевих моделей,
до проведення експериментів та аналізу результатів. Додатково досліджено можливості використання ІІ-агентів для автоматизації рутинних задач у науковому дослідженні.

Основні результати та висновки дослідження можна узагальнити наступним чином.

**1. Систематизовано сучасний стан методів геолокації зображень.** У процесі підготовки огляду літератури проаналізовано понад 30 наукових публікацій, що охоплюють еволюцію підходів до задачі від ранніх методів на основі ручних ознак (SIFT, SURF) до сучасних контрастивних мультимодальних моделей (CLIP, GeoCLIP, StreetCLIP). Виявлено ключову тенденцію переходу від класифікаційного підходу з дискретними географічними комірками до неперервної геолокації через вирівнювання мультимодальних ембедингів у спільному латентному просторі. Встановлено, що доменно-специфічне попереднє навчання на геотегованих зображеннях є одним із найефективніших підходів для підвищення якості геолокації.

**2. Зібрано та підготовлено спеціалізований датасет вуличних зображень.** Сформовано датасет обсягом 35 019 зображень із восьми міст (п'ять українських: Київ, Львів, Одеса, Харків, Дніпро; три центральноєвропейських: Варшава, Прага, Будапешт) з платформ OpenStreetView 5M (OSV-5M) та Mapillary під ліцензією CC-BY-SA. Реалізовано географічно стратифікований поділ даних з використанням гексагональної індексації H3 на рівні 4, що забезпечує відсутність просторового витоку між тренувальним, валідаційним та тестовим наборами. Застосовано збалансовану випадкову вибірку для компенсації нерівномірного розподілу зображень між містами.

**3. Розроблено та реалізовано три нейромережеві архітектури для визначення геолокації.** Імплементовано та досліджено три принципово різні підходи:

- *BaselineCNN* на основі EfficientNet-B2 з трансферним навчанням від ImageNet — класичний підхід, що досягнув Top-1 Accuracy 72,3% та середнього GeoScore 4116;
- *StreetCLIP Fine-tuning* з використанням доменно-специфічної моделі StreetCLIP (CLIP-ViT-L-14-336) та лінійного зонду — найкращий результат класифікації з Top-1 Accuracy 81,7% та GeoScore 4443;
- *GeoCLIP-inspired model* з контрастивним навчанням та кодувальником GPS на базі Random Fourier Features — найкращий результат тонкозернистої локалізації з медіанною відстанню 18,9 км та GeoScore 4527.

**4. Найкращим класифікаційним результатом є StreetCLIP Fine-tuning з Top-1 Accuracy 81,7%.** Ця модель перевершила базову CNN на 9,4 процентних пунктів, що кількісно підтверджує ефективність доменно-специфічного попереднього навчання для задачі геолокації. Дослідження абляцій показало, що часткове розморожування останніх двох шарів ViT додатково підвищує точність до 83,2%. Порівняння StreetCLIP зі стандартним CLIP (76,8%) підтверджує, що різниця у 4,9 п.п. повністю пояснюється адаптацією до геолокаційного домену.

**5. GeoCLIP-inspired модель досягла найкращої тонкозернистої локалізації.** Незважаючи на дещо нижчу точність класифікації за містами (78,4%), GeoCLIP продемонструвала найкращі результати у метриках геодезичної відстані: медіанна відстань 18,9 км (порівняно з 22,7 км для StreetCLIP та 48,2 км для BaselineCNN) та найвищий середній GeoScore 4527. Модель здатна локалізувати 3,8% зображень з точністю до 1 км, що принципово неможливо для класифікаційних підходів. Це підтверджує перспективність контрастивного підходу з неперервною GPS-репрезентацією для задач, де потрібна точна географічна локалізація.

**6. Проведено детальний аналіз помилок моделей.** Виявлено, що основними джерелами помилок є: візуальна подібність промислової та радянської забудови (особливо пара Харків–Дніпро), периферійні райони міст з низькою архітектурною виразністю, автомагістралі та промислові зони без характерних маркерів, а також зимові зображення, де сніг маскує географічно інформативні ознаки. Візуалізація Grad-CAM підтвердила, що моделі навчаються використовувати архітектурні елементи, дорожні маркування, вуличні написи та тип рослинності як географічні маркери.

**7. Розроблено агентний пайплайн на базі OpenClaw для автоматизації дослідницького процесу.** Створено мультиагентну систему з чотирма спеціалізованими агентами (дослідницький, кодовий, даних та написання), що забезпечила скорочення часу виконання рутинних задач на 70–90% для технічних завдань та на 50–65% для творчих завдань. Визначено чіткий розподіл відповідальності між ІІ-агентом (виконання, пошук, обчислення) та дослідником (рішення, валідація, творчість, етика).

**8. Сформульовано практичні рекомендації для розгортання моделей геолокації.** На основі проведених експериментів визначено оптимальні стратегії для різних сценаріїв використання:

- *Класифікація за містами з максимальною точністю:* StreetCLIP з розмороженими останніми двома шарами ViT (83,2% Top-1 Accuracy).
- *Класифікація за містами з мінімальними ресурсами:* замороженний StreetCLIP + лінійний зонд (81,7% Top-1, лише 6K навчальних параметрів, навчання за 2 години).
- *Тонкозерниста координатна локалізація:* GeoCLIP з GPS-галереєю 1000–5000 точок (медіанна відстань 15–19 км).
- *Висока пропускна здатність:* BaselineCNN (3,1 мс/зображення, Top-1 72,3%).
- *Каскадний підхід:* GeoCLIP для визначення GPS-координат + StreetCLIP для верифікації міста — поєднує переваги обох підходів.

Для продуктивного розгортання рекомендується конвертація моделей у формат ONNX або TorchScript з подальшою оптимізацією через NVIDIA TensorRT, що може зменшити час інференсу на 40–60%. Для виробничого застосування рекомендується використовувати StreetCLIP для задач класифікації за містами та GeoCLIP для задач точної координатної локалізації. Оптимальним є каскадний підхід: GeoCLIP для визначення координат із подальшим StreetCLIP для верифікації на рівні міста. Розмір GPS-галереї для GeoCLIP рекомендовано встановлювати на рівні 1000–5000 точок залежно від вимог до точності та часу відповіді.

**9. Проведено статистичний аналіз результатів.** Усі виявлені відмінності між моделями підтверджено статистично: тест МакНемара показав значущість на рівні p < 0,001 для всіх основних порівнянь. Бутстреп-аналіз довірчих інтервалів підтвердив відсутність перетину інтервалів Top-1 Accuracy для BaselineCNN та StreetCLIP, що свідчить про надійність виявлених відмінностей.

**10. Досліджено вплив факторів середовища на якість геолокації.** Встановлено, що точність суттєво залежить від типу міського середовища (89,3% для історичних центрів vs 61,2% для промислових зон) та сезону (84,2% для літніх зображень vs 73,4% для зимових). Ці результати мають практичне значення для розуміння меж застосовності моделей та планування збору додаткових даних.

**Напрямки подальших досліджень.** На основі отриманих результатів та виявлених обмежень визначено наступні перспективні напрямки продовження роботи:

1. *Масштабування до рівня районів* — перехід від класифікації за вісьмома містами до класифікації за адміністративними районами (50–100 класів), що потребує збільшення датасету та адаптації архітектур.

2. *Мультимодальний підхід* — інтеграція візуальної інформації з текстовою (OCR написів на зображеннях), метаданими (час доби, пора року, напрямок камери) та картографічними даними для покращення точності.

3. *Часовий аналіз* — дослідження стабільності моделей у часі (temporal drift) та розробка методів адаптації до змін у міському ландшафті, включаючи зміни внаслідок збройного конфлікту.

4. *Розгортання як веб-сервісу* — створення REST API для інференсу в реальному часі, оптимізація моделі через квантизацію (INT8) та дистиляцію знань для зменшення часу відповіді, побудова веб-інтерфейсу для інтерактивної демонстрації.

5. *Дослідження послідовностей зображень* — використання просторово-часового контексту послідовних кадрів вуличних камер для підвищення стабільності та точності локалізації.

6. *Розширення географічного покриття* — включення додаткових українських та європейських міст, а також сільських територій для створення більш універсальної моделі геолокації.

7. *Ансамблевий підхід* — поєднання передбачень класифікаційної та контрастивної моделей через навчальний ансамбль (stacking), що може поєднати високу точність класифікації StreetCLIP з тонкозернистою локалізацією GeoCLIP.

8. *Адаптація до нових міст без перенавчання (few-shot learning)* — розробка методів адаптації навченої моделі до нових міст з використанням мінімальної кількості прикладів (5–50 зображень на місто), що є критичним для масштабування системи.

9. *Дослідження впливу якості зображень* — систематичний аналіз робастності моделей до деградації якості зображень (розмиття, стиснення JPEG, низька роздільна здатність), що є типовим для реальних умов використання камер відеонагляду та мобільних пристроїв.

Результати даного дослідження підтверджують, що сучасні нейромережеві підходи, зокрема мультимодальні контрастивні моделі з доменно-специфічним попереднім навчанням, є ефективним інструментом для визначення геолокації вуличних зображень і мають значний потенціал для практичного застосування у сферах геоінформаційних систем, містобудівного аналізу, верифікації візуального контенту та цифрової картографії.

Практична цінність отриманих результатів полягає у кількох аспектах. По-перше, розроблена методологія збору та підготовки датасету вуличних зображень з географічно стратифікованим поділом може бути використана для побудови аналогічних датасетів для інших регіонів. По-друге, порівняльний аналіз трьох архітектур надає практичні рекомендації щодо вибору моделі залежно від конкретної задачі: StreetCLIP для класифікації за містами, GeoCLIP для точної координатної локалізації. По-третє, досвід використання ІІ-агентів у дослідницькому процесі задокументовано з аналізом ефективності та обмежень, що може слугувати орієнтиром для інших дослідників, які розглядають можливість інтеграції ІІ-інструментів у свою роботу.

Підсумовуючи, слід наголосити на міждисциплінарному характері проведеного дослідження, яке поєднує методи глибокого навчання (трансферне навчання, контрастивне навчання, мультимодальні моделі), геоінформатики (H3-індексація, проєкція Equal Earth, формула Гаверсина), програмної інженерії (Docker, MLflow, Git-версіонування) та штучного інтелекту (агентні системи, автоматизація дослідницьких пайплайнів). Такий міждисциплінарний підхід є типовим для сучасних досліджень на перетині комп'ютерного зору та геопросторового аналізу.

Окремо варто підкреслити актуальність дослідження для України. Створений датасет вуличних зображень п'яти українських міст є, наскільки нам відомо, першим спеціалізованим датасетом такого роду для задачі геолокації. Результати дослідження можуть бути використані для: верифікації геолокації фотографій у журналістських розслідуваннях та OSINT-аналізі; аналізу стану міської інфраструктури за даними вуличних камер; побудови інтелектуальних навігаційних систем, що доповнюють GPS-дані візуальною інформацією; моніторингу змін у міському ландшафті з часом.

Таким чином, дипломна робота вносить внесок як у прикладну сферу геолокації зображень (порівняльний аналіз архітектур, спеціалізований датасет українських міст), так і у методологічну сферу ІІ-асистованого наукового дослідження (документація практик використання агентних систем у дослідницькому процесі, аналіз розподілу відповідальності між людиною та ІІ).
